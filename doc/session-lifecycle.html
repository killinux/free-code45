T1: 轮次准备
Local queryLoop() 开始新一轮

  ├ Memory startRelevantMemoryPrefetch()  ← 异步启动, 非阻塞

  │   ├ Local 批量读取 frontmatter (name, description, type)
  │   └ LLM Sonnet sideQuery: 选 ≤5 个最相关记忆

  │
  ├ Prompt 组装 system prompt
  │   └ appendSystemContext(systemPrompt, systemContext)
  │     systemContext = Phase 1 缓存的 git status
  │
  ├ Context 组装 messages

  ├ 未讨论 计算 attachments (技能发现, 延迟工具, 嵌套记忆)
  │
  └ Prompt 组装 tools 参数

T3: LLM 流式响应
LLM 模型推理 → SSE 流式事件

  ├ message_start      → 初始化 usage, model
  ├ content_block_start → 新 block 开始 (text / tool_use / thinking)
  ├ content_block_delta → 增量数据流
  │   ├ text_delta         → 文本片段 (渲染到终端)
  │   ├ input_json_delta   → 工具输入 JSON 片段 (拼接到 input 字符串)
  │   └ thinking_delta     → 思维链片段
  ├ content_block_stop  → block 完成
  │   └ tool_use block: normalizeContentFromAPI() 解析 input JSON
  ├ message_delta       → stop_reason + final usage
  └ message_stop

  最终 assistant message.content 可能包含:
  [ text, tool_use, tool_use, thinking, text, tool_use, ... ]
↓
Phase 3: 工具执行循环 (Tool Loop) — 如果 stop_reason == 'tool_use'
T4: 工具提取与调度
Local query.ts:829

  ├ Tool 提取 ToolUseBlock[]
  │   └ content.filter(c => c.type === 'tool_use')
  │     每个 block: { type, id, name, input }
  │
  ├ Tool toolOrchestration.ts: runTools()
  │   ├ 按 isConcurrencySafe() 分批
  │   ├ 并发安全工具 → runToolsConcurrently() (上限 10)
  │   │   Read, Glob, Grep, WebSearch 等可并发
  │   └ 非并发安全 → runToolsSerially()
  │       Bash, Edit, Write 等需串行
  │
  ├ 未讨论 StreamingToolExecutor (可选)
  │   └ FGTS 模式: 边接收 input_json_delta 边开始执行
  │
  └ 对每个工具调用 runToolUse()
↓
T5: 单工具执行 (per tool_use block)
Local toolExecution.ts: runToolUse()

  Step 1: 输入验证
  ├ inputSchema.safeParse(input)  ← Zod schema 验证
  └ tool.validateInput()          ← 工具自定义验证 (可选)
      如果失败 → 返回 error tool_result, 跳过执行

  Step 2: 权限检查
  ├ Hook runPreToolUseHooks()  ← 用户自定义 hooks
  │   └ hooks 可返回: 允许 / 拒绝 / 修改 input / 停止
  ├ canUseTool() → hasPermissionsToUseTool()
  │   ├ 检查规则: alwaysAllow / alwaysDeny / alwaysAsk
  │   ├ 规则来源: globalSettings, localSettings, projectSettings, flagSettings
  │   └ 如果 ask → 展示权限对话框等待用户确认
  └ 如果拒绝 → 返回 error tool_result, 记录 tengu_tool_use_rejected

  Step 3: 工具执行
  ├ Tool tool.call(input, context, canUseTool, assistantMessage, onProgress)
  │   实际执行本地操作: shell 命令 / 文件读写 / 搜索 / MCP 调用 / ...
  └ 返回 ToolResult { data, newMessages?, contextModifier?, mcpMeta? }

  Step 4: 结果处理
  ├ tool.mapToolResultToToolResultBlockParam(data, toolUseID)
  │   └ { type: 'tool_result', tool_use_id, content, is_error }
  ├ applyToolResultBudget()  ← 超过 maxResultSizeChars 则落盘
  ├ Hook runPostToolUseHooks()  ← 可修改结果
  └ 加入 toolResults[] → 下一次 API 调用的 messages
↓
T6: 循环决策
Local query.ts

  ├ Memory 检查 pendingMemoryPrefetch 是否完成
  │   ├ 如果完成: 读取选中的记忆文件内容 + 过期警告
  │   └ 作为 attachment 加入 toolResults[]
  │     注意: 这些记忆内容在下一次 T2 API 调用时 LLM 才看到
  │
  ├ 未讨论 检查 auto-compaction 触发条件
  │   └ 如果 token 接近上限 → 自动压缩对话历史
  │
  ├ 检查 stop_reason:
  │   ├ 'tool_use'   → 还有更多工具要调用, 回到 T1
  │   ├ 'end_turn'   → LLM 完成回答, 进入 T7
  │   ├ 'max_tokens' → 输出截断, 可能自动恢复
  │   └ budget exceeded / max turns → 强制结束
  │
  └ 如果继续循环: 回到 T1 (Context/systemPrompt 不变, 仅 messages 更新)
↓

核心概念交互关系

所有概念如何连接在一起

概念交互矩阵
A → B System Prompt Context Memory Tool Use Hooks
System Prompt — systemContext 追加到末尾 MEMORY.md 索引嵌入 memory section 定义工具使用规则和风格 告诉 LLM hooks 存在
Context CLAUDE.md 作为 messages[0] 注入 — CLAUDE.md 和 MEMORY.md 分离但互补 CLAUDE.md 影响 LLM 工具调用行为 —
Memory 记忆指令嵌入 system prompt /remember 可提升记忆到 CLAUDE.md — LLM 用 Write/Edit 写记忆; 用 Grep 验证记忆 —
Tool Use tool schema 通过 tools 参数传递 工具修改文件不刷新 Context 缓存 工具写入新记忆 → 下轮 Sonnet 可选 — Pre/PostToolUse hooks 包裹每次执行
Hooks — — — 可修改工具 input/output, 控制权限 —
