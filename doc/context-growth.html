Context 增长解剖

claude-hud 的 Context% 增长时，内部的 Context、Memory、Prompt、Tool Result 各自干了什么

两个 "Context" 概念对齐
claude-hud 的 Context% = 整个 API 请求的 token 总量 / 模型上下文窗口大小

它包含了我们讨论过的所有概念:

┌──────────────────────────────────────────────────────────────────┐
│  HUD Context% = System Prompt + Tools Schema + Messages         │
│                                                                │
│  其中 System Prompt 包含:                                      │
│    • 静态行为指令 (角色/风格/安全/工具规则)                     │
│    • Memory 指令 + MEMORY.md 索引                              │
│    • 环境信息 (cwd, model, date)                               │
│    • git status (我们讨论的 "System Context")                  │
│                                                                │
│  其中 Messages 包含:                                            │
│    • CLAUDE.md 内容 (我们讨论的 "User Context")                │
│    • 用户消息                                                 │
│    • LLM 回复 (text + thinking)                               │
│    • tool_result (工具执行结果 — 主要增长源)                 │
│    • Memory 附件 (Sonnet 选出的相关记忆内容)                  │
│    • system-reminder 附件 (技能/延迟工具等)                    │
└──────────────────────────────────────────────────────────────────┘

增长公式

Context% 到底是怎么算的

HUD Context% =

  ( system_prompt_tokens    ~6,000-10,000  固定，含记忆指令+MEMORY.md索引+git status
  + tools_schema_tokens     ~8,000-12,000  固定，43 个工具定义
  + claude_md_tokens        ~500-3,000     固定 (memoized)
  + conversation_tokens     线性增长，每轮 +300-5,000
  + tool_result_tokens      最大变量，每次工具调用 +100-10,000
  + memory_attachment_tokens 波动，每轮 0 或 500-3,000
  )
  ÷ context_window_size     200,000 (默认) 或 1,000,000 (1m beta)
  × 100%

增长速度排序:

  tool_result >> 对话历史 > 记忆附件 >> system prompt = tools = context (不变)

关键洞察:

  • 固定开销 (system prompt + tools + CLAUDE.md) 约占 ~10% (200K 窗口)
  • Context 和 Memory 索引是固定开销，不是增长源
  • tool_result 是增长的主要驱动力（一次 Read 大文件就可能 +5%）
  • 对话文本本身增长相对缓慢
  • Memory 附件每轮波动，但单轮量不大 (~1-2%)
  • Prompt Cache 不减少 Context%，只减少费用（cache_read 按正常费率的 10% 计）
控制增长的机制
机制触发条件效果
maxResultSizeChars 单个 tool_result 过大 超限部分落盘，只发送预览 + 文件路径。防止单次工具调用撑爆窗口
Function Result Clearing (FRC) 系统自动 清理历史轮次的 tool_result 内容，只保留摘要。LLM 被告知内容已清理
Auto-Compaction Context% 接近上限 将对话历史压缩为摘要。Context% 降至 ~10-15%。触发 CLAUDE.md 缓存刷新
/compact 命令 用户手动 同 Auto-Compaction + 重置 CLAUDE.md memoize 缓存
Prompt Caching 自动 不减少 Context%，但减少费用。system prompt + tools schema 首次缓存，后续 cache hit
1M Context Beta 模型支持 + beta flag context_window_size 从 200K 升到 1M。相同 token 量下 Context% 降到 1/5
总结: claude-hud 的 Context% 是整个 API 请求的 token 占用率。我们讨论的 "Context" (CLAUDE.md + git status) 只是其中一个固定开销组件，占 ~1-2%，不随对话增长。真正让 Context% 增长的主力是 tool_result（工具执行结果的累积），其次是对话历史（用户消息 + LLM 回复）。Memory 索引是固定开销；Memory 附件每轮波动 0-2%。当 Context% 接近上限时，auto-compaction 会触发压缩，这是唯一会刷新 CLAUDE.md 缓存的会话内事件。
