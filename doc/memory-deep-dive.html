路径解析优先级

getAutoMemPath() 按以下优先级确定记忆目录：

  1. CLAUDE_COWORK_MEMORY_PATH_OVERRIDE 环境变量（Cowork 场景完整路径覆盖）
  2. settings.json 中的 autoMemoryDirectory（仅信任 policy/flag/local/user 来源，不信任 projectSettings）
  3. 默认路径：~/.claude/projects/<sanitized-git-root>/memory/
Git Worktree 共享：使用 findCanonicalGitRoot() 确保所有 worktree 共享同一个记忆目录。无 Git 仓库时降级为 getProjectRoot()。

getAutoMemPath() 在 getProjectRoot() 上做了记忆化（memoize），避免频繁的 stat 调用。

2. 四种记忆类型

src/memdir/memoryTypes.ts

user — 用户画像

始终私有

记录用户的角色、目标、职责和知识背景。帮助未来对话中针对性地调整回答风格。

何时保存

了解到用户角色、偏好、职责或知识领域的任何细节时。

如何使用

根据用户画像调整回答深度和方式。例如：资深工程师 vs 初学者需要不同的解释方式。

feedback — 行为反馈

默认私有，项目级约定可共享

用户对工作方式的纠正和确认。同时记录失败和成功 — 仅记录纠正会导致过度谨慎。

何时保存

用户纠正做法（"不要这样"、"停止 X"）或确认非显而易见的做法（"正是这样"、"完美"）时。

结构要求

规则本身
**Why:** 用户给出的原因
**How to apply:** 何时/何处应用此规则

project — 项目上下文

私有或团队，偏向团队

正在进行的工作、目标、截止日期、事故等项目级信息，不可从代码或 Git 历史中直接推导。

何时保存

了解到谁在做什么、为什么、何时完成。必须将相对日期转为绝对日期（"周四" → "2026-03-05"）。

结构要求

事实或决定
**Why:** 动机（约束、截止日期、利益相关者需求）
**How to apply:** 如何影响建议

reference — 外部引用

通常共享

指向外部系统中信息所在位置的指针。帮助在未来需要时知道去哪里查找。

何时保存

了解到外部系统中的资源及其用途时。例如：Bug 跟踪在 Linear 的 "INGEST" 项目中。

记忆文件格式（Frontmatter）

---
name: {{记忆名称}}
description: {{一行描述 — 用于未来会话中判断相关性，所以要具体}}
type: {{user | feedback | project | reference}}
---

{{记忆内容 — feedback/project 类型建议使用: 规则/事实 + Why: + How to apply: 结构}}

不应保存的内容

  • 代码模式、架构、文件路径、项目结构 — 可从代码直接推导
  • Git 历史、最近变更 — git log / git blame 是权威来源
  • 调试方案、修复食谱 — 修复在代码中，上下文在 commit message 中
  • 已在 CLAUDE.md 中记录的内容
  • 临时任务详情、进行中的工作、当前会话上下文
关键判断：即使用户要求保存 PR 列表或活动摘要，也应该询问"什么是令人惊讶的或非显而易见的？" — 那部分才值得保留。

3. 记忆扫描与相关性选择

文件扫描 scanMemoryFiles()

src/memdir/memoryScan.ts

参数值说明
MAX_MEMORY_FILES200最多返回的文件数
FRONTMATTER_MAX_LINES30每个文件读取的最大行数（仅解析 frontmatter）

算法

  1. 递归扫描记忆目录，筛选 .md 文件（排除 MEMORY.md 本身）
  2. 使用 Promise.allSettled() 并行读取所有文件的 frontmatter
  3. 单次 I/O：readFileInRange() 内部同时完成 stat（获取 mtimeMs）和读取，避免双重 stat
  4. 提取 description 和 type 字段
  5. 按修改时间降序排列（最新优先）
  6. 返回前 200 个 MemoryHeader：{filename, filePath, mtimeMs, description, type}

相关性选择 findRelevantMemories()

src/memdir/findRelevantMemories.ts

算法

  1. 调用 scanMemoryFiles() 获取所有记忆头信息
  2. 过滤掉已展示过的路径（alreadySurfaced）
  3. 将清单（manifest）格式化为文本列表：[type] filename (ISO时间): description
  4. 调用 Sonnet 3.5 模型，传入用户查询 + 清单 + 最近使用的工具列表
  5. 模型返回最多 5 个最相关的文件名（JSON Schema 约束输出）
  6. 验证返回的文件名是否在实际清单中存在
工具过滤逻辑：如果用户正在使用某个工具，不展示该工具的使用文档（已在上下文中）。但仍然展示关于该工具的警告/注意事项（正在使用时更需要知道）。

记忆注入到对话中

src/utils/attachments.ts

限制值说明
每文件最大行数200注入时截断
每文件最大字节4,096约 4KB
每轮最多文件数5每轮最多 ~20KB

预取机制 startRelevantMemoryPrefetch()

  1. 提取最后一条真实用户消息（跳过系统注入的 meta 消息）
  2. 要求多词提示（正则检查包含空格）
  3. 创建子 AbortController（与主轮次 abort 绑定）
  4. 非阻塞启动异步获取（后台运行）
  5. 返回 Disposable 句柄（using 语法，所有退出路径自动释放）
  6. 消费点在 query.ts 中：如果已完成则读取，否则跳过 — 永远不阻塞主轮次

记忆新鲜度标记

src/memdir/memoryAge.ts

  • memoryAge(mtimeMs) → 'today' / 'yesterday' / 'N days ago'（人类可读，模型表现更好）
  • memoryAgeDays(mtimeMs) → 向下取整天数，时钟偏差时钳位到 0
  • 新鲜记忆（≤1 天）：无警告
  • 陈旧记忆（>1 天）：追加 <system-reminder> 标签包裹的警告： "此记忆已有 N 天。记忆是时间点观察，在引用前请验证当前状态。"
验证义务：在向用户推荐基于记忆的信息前：文件路径 → 检查是否存在；函数名 → grep 查找；准备执行（而非仅讨论历史）→ 必须先验证。"记忆说 X 存在" ≠ "X 现在存在"。

4. 会话记忆（Session Memory）原理

src/services/SessionMemory/sessionMemoryUtils.ts · src/services/SessionMemory/prompts.ts

状态跟踪变量

变量说明
lastSummarizedMessageId最后一次提取处理到的消息 UUID
tokensAtLastExtraction上次提取时的上下文大小（用于计算增量）
extractionStartedAt提取开始时间戳（防止并发）
sessionMemoryInitialized是否已过初始化阈值

触发决策算法 shouldExtractMemory()

1. 检查初始化阈值
   if (!initialized && tokens < 10,000) return false
   else markInitialized()

2. 检查增量阈值
   hasMetTokenThreshold = (currentTokens - tokensAtLastExtraction) >= 5,000

3. 检查工具调用阈值
   hasMetToolCallThreshold = toolCallsSinceLastUpdate >= 3

4. 检查最后一轮是否有工具调用
   hasToolCallsInLastTurn = 最后一条 assistant 消息是否包含 tool_use

5. 决策
   shouldExtract = (token阈值 AND 工具调用阈值)
                OR (token阈值 AND 最后一轮无工具调用)

   注意: Token 阈值是必要条件，永远不可跳过
设计意图："最后一轮无工具调用" 表示对话到了自然断点（模型给出了文本回复而非继续工具操作），此时提取最安全，不会打断正在进行的工作。

截断控制

  • 每个分区上限：2,000 token
  • 总计上限：12,000 token

  • 模板中的分区标题和 斜体描述 必须完整保留，仅编辑描述下方的内容

自定义模板

用户可在以下路径放置自定义文件：

  • ~/.claude/session-memory/config/template.md — 自定义笔记模板
  • ~/.claude/session-memory/config/prompt.md — 自定义提取提示词

5. 持久记忆提取（Auto Memory）原理

src/services/extractMemories/extractMemories.ts · src/services/extractMemories/prompts.ts

闭包作用域状态

initExtractMemories() 创建一个闭包，所有可变状态封装在内部：

  • lastMemoryMessageUuid — 游标，每次提取只处理新增消息
  • inProgress — 布尔锁，防止重叠运行
  • turnsSinceLastExtraction — 合格轮次计数器，重置为 0 after each run
  • pendingContext — 暂存的上下文（在运行中有新请求时）
  • inFlightExtractions — Set<Promise>，用于 drain 等待

互斥与重叠保护

新请求到达
    |
    v
inProgress? ----YES----> 暂存到 pendingContext（覆盖式，只保留最新）
    |                         |
    NO                    当前运行完成
    |                         |
    v                         v
执行 runExtraction()      有暂存? ---YES--> 执行尾随提取（trailing run）
    |                                       （游标已前进，只处理增量）
    v
提取完成 --> 检查 pendingContext
主代理写入互斥：hasMemoryWritesSince() 扫描 assistant 消息中的 Write/Edit 工具调用，如果目标路径在 isAutoMemPath() 内，说明主代理已直接写入记忆，此时跳过后台提取并前移游标。

提取提示词设计

src/services/extractMemories/prompts.ts

关键指令

  • 指定要分析的消息数量：~${newMessageCount} messages
  • 列出现有记忆文件清单 — 指示更新已有文件而非创建重复
  • 限制工具使用：Read/Grep/Glob + 只读 Bash + Edit/Write 仅限记忆目录
  • 高效轮次预算：第 1 轮全部 Read，第 2 轮全部 Write（并行化）
  • 禁止代码验证：仅使用对话中的内容，不去检查代码库

两步保存流程

  1. 将记忆写入独立文件（如 user_role.md），使用 frontmatter 格式
  2. 在 MEMORY.md 中添加索引条目（一行，~150 字符）

团队记忆变体

当 feature('TEAMMEM') 启用时，使用 buildExtractCombinedPrompt()，区分私有/团队记忆目录，每种类型附带 scope 指导。

提取后的可见反馈

如果子代理写入了记忆文件（排除 MEMORY.md 索引更新），通过 appendSystemMessage(createMemorySavedMessage(paths)) 在 REPL 中显示一条系统消息，让用户知道记忆已保存。

6. Auto Dream（记忆合并）

src/services/autoDream/

Auto Dream 是一个后台记忆合并机制，类似于人类睡眠期间的记忆整理。它会将多个会话中积累的零散记忆合并、去重、精简。

触发门控（从低成本到高成本）

  1. 时间门控：距上次合并 ≥ minHours（默认 24 小时）
  2. 会话门控：自上次合并以来 ≥ minSessions 个新会话（默认 5 个，排除当前会话）
  3. 锁门控：没有其他进程正在合并

扫描节流

当时间门控通过但会话门控未通过时，会话扫描会被节流 10 分钟，防止重复的 lock-stat 调用。

锁机制

tryAcquireConsolidationLock() 通过原子文件 mtime 更新实现锁。支持回滚（进程被杀时恢复到先前 mtime）。

KAIROS 模式的特殊路径

在 KAIROS 模式（永续助手会话）下，使用基于日志的模型：

  • 每天追加日志到 <memdir>/logs/YYYY/MM/YYYY-MM-DD.md
  • 每晚通过 /dream 技能将日志合并为主题文件 + 更新 MEMORY.md
  • 日志缓存模式：模板中不包含字面日期，模型从 currentDate 附件推导（午夜翻转时自动更新）

7. 团队记忆安全机制

src/memdir/teamMemPaths.ts

路径遍历防护

sanitizePathKey() 拒绝的模式

  • Null 字节
  • URL 编码的遍历：%2e%2e%2f = ../
  • Unicode 标准化的遍历：全角 . 和 /
  • 反斜杠（Windows 路径分隔符）
  • 绝对路径

validateTeamMemWritePath() 两阶段验证

  1. 字符串级：检查路径前缀包含关系
  2. 符号链接解析：realpathDeepestExisting() 沿目录树向上解析符号链接，确保实际文件系统位置仍在团队记忆目录内
前缀攻击防护：要求前缀匹配后紧跟路径分隔符。/foo/team-evil/ 不会匹配 /foo/team/。

8. CLAUDE.md 与记忆系统的交互

src/utils/claudemd.ts

文件加载优先级（低到高）

  1. Managed：/etc/claude-code/CLAUDE.md（全局管理策略）
  2. User：~/.claude/CLAUDE.md（用户全局偏好）
  3. Project：项目根目录的 CLAUDE.md、.claude/CLAUDE.md、.claude/rules/*.md
  4. Nested：从 CWD 到 git root 每一级目录的 CLAUDE.md
  5. Local：CLAUDE.local.md（最高优先级，不提交到仓库）

@include 指令

  • 语法：@path、@./relative、@~/home、@/absolute
  • 裸 @path 等价于 @./path
  • 仅在叶文本节点中生效（不在代码块中）
  • 内置循环引用防护（通过已处理文件集合追踪）
  • 不存在的文件静默忽略
  • 被包含的文件作为单独条目插入到包含者之前

HTML 注释剥离

stripHtmlComments() 使用 marked 词法分析器进行块级检测：

  • 剥离 <!-- ... --> 块注释
  • 保留代码块内的注释
  • 保留未闭合的注释（避免误删有拼写错误的内容）

嵌套记忆附件

getNestedMemoryAttachments() 在以下场景触发：

  • 工具操作了某个文件 → 检查该文件路径是否有关联的条件规则
  • 从 CWD 到目标文件路径的每一级目录中查找 CLAUDE.md
  • paths frontmatter 字段定义条件匹配模式（如 src/**/*.ts）
