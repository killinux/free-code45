getContextWindowForModel() — 窗口大小判定逻辑

1M 上下文禁用：CLAUDE_CODE_DISABLE_1M_CONTEXT 环境变量可全局禁用 1M 上下文，用于 HIPAA 合规场景。

2. 查询管线（Query Pipeline）

src/query.ts

这是整个上下文管理的核心编排文件。每次模型调用前，消息经过多个处理阶段：

1
提取压缩边界后的消息
getMessagesAfterCompactBoundary(messages) — 从最近一次压缩边界标记之后开始
↓
2
工具结果预算控制
applyToolResultBudget() — 对每条消息中过大的工具结果进行截断
↓
3
历史裁剪（Snip）
snipCompactIfNeeded() — 移除中间轮次的旧消息，释放 token
↓
4
微压缩
microcompactMessages() — 清理旧的工具结果内容（时间触发 / 缓存编辑）
↓
5
上下文折叠
applyCollapsesIfNeeded() — 将已提交的折叠摘要投射到消息视图上
↓
6
自动压缩
autoCompactIfNeeded() — 当 token 超过阈值（~93%）时触发完整压缩
↓
7
阻塞限制检查
如果 token 达到 ~98%（effectiveWindow - 3K），直接阻塞并提示用户
↓
8
组装 API 请求
prependUserContext() 前置用户上下文
appendSystemContext() 拼接系统上下文
toolToAPISchema() 转换工具定义
最终调用 callModel()
设计原则：各阶段从轻到重递进 — 先裁剪（零开销）、再微压缩（低开销）、再折叠（中开销）、最后完整压缩（高开销）。如果轻量级手段就能让 token 回落到阈值以下，则跳过重量级手段。

3. 消息管理与规范化

src/utils/messages.ts

消息类型

类型说明
UserMessage用户输入（可包含文本、工具结果、图片等内容块）
AssistantMessage模型回复（包含 usage 数据、message.id 等元信息）
AttachmentMessage附件（文件、任务、记忆、IDE 选区等）
SystemMessage系统消息（进度提示、错误、压缩边界标记等）
HookResultMessage钩子执行结果
TombstoneMessage已删除消息的占位标记

normalizeMessagesForAPI() — API 规范化

将内部 Message[] 转换为 API 兼容格式，是发送给模型前的最后一步处理：

  • 合并同源记录：将共享同一 message.id 的多条 assistant 记录合并（流式输出的拆分产物）
  • 剥离 advisor 块：stripAdvisorBlocks() 移除顾问模型的内容
  • 剥离 caller 字段：stripCallerFieldFromAssistantMessage() 移除 tool_use 中的 caller 标记
  • 剥离 tool_reference 块：stripToolReferenceBlocksFromUserMessage()
  • 确保配对完整：ensureToolResultPairing() 保证每个 tool_use 都有对应的 tool_result

压缩边界标记

createCompactBoundaryMessage() 创建一种特殊的系统消息，标记压缩发生的位置。查询管线通过 getMessagesAfterCompactBoundary() 从最近边界之后开始取消息，确保压缩后的摘要和新消息构成完整上下文。

消息分组 — API 轮次边界

src/services/compact/grouping.ts

groupMessagesByApiRound() 以 assistant 消息的 message.id 变化为界，将消息分组。

  • 同一 API 响应的流式块共享 message.id，归入同一组
  • message.id 改变意味着新的 API 轮次开始
  • API 约定保证每个 tool_use 在下一个 assistant 轮次前被解决，因此边界天然安全
  • 对于畸形会话（如恢复/截断后出现悬空的 tool_use），由 ensureToolResultPairing() 在 API 调用时修复

4. 系统提示词构造

src/constants/systemPromptSections.ts · src/utils/api.ts

分段缓存架构

系统提示词被拆分为多个可缓存的 section：

  • systemPromptSection(name, computeFn) — 缓存段，计算一次后复用
  • DANGEROUS_uncachedSystemPromptSection(name, computeFn, reason) — 不可缓存段（每次重新计算，会导致缓存失效）
  • resolveSystemPromptSections(sections) — 异步解析所有段并拼接
  • clearSystemPromptSections() — 在 /clear 或 /compact 时清空缓存

动态边界分割

通过 splitSysPromptPrefix() 将系统提示词在 SYSTEM_PROMPT_DYNAMIC_BOUNDARY 处分割：

  • 静态前缀：附加 cache_control（ephemeral + TTL + scope），实现跨请求缓存命中
  • 动态后缀：无缓存控制，每次请求可能不同（如实时状态信息）

上下文拼接

函数位置说明
appendSystemContext() 系统提示词尾部 拼接系统上下文（如 Git 状态、当前日期等）
prependUserContext() 消息列表头部 前置用户上下文（CLAUDE.md 内容、内存文件等），作为第一条用户消息

5. Prompt 缓存机制

src/services/api/claude.ts · src/services/api/promptCacheBreakDetection.ts

缓存控制结构

{
  type: 'ephemeral',
  ttl?: '1h',               // 1 小时 TTL（默认 5 分钟）
  scope?: 'global' | 'org'  // 缓存范围
}

缓存控制被应用于：

  • 系统提示词静态前缀 — 分割后的不变部分
  • 工具定义 — 如果启用全局缓存范围

缓存中断检测

系统持续跟踪可能导致缓存失效的状态变化：

跟踪的状态参数

  • systemHash — 系统提示词哈希
  • toolsHash — 全部工具定义哈希
  • perToolHashes — 单个工具哈希
  • cacheControlHash — 缓存控制标记
  • model — 模型名称
  • fastMode — 快速模式开关
  • betas — Beta 头部列表
  • effortValue — 推理努力等级

检测常量

2K MIN_CACHE_MISS_TOKENS（触发警告的最小 token 数）
5 分钟 默认缓存 TTL
1 小时 实验性长 TTL

关键函数

  • recordPromptState() — 每次 API 调用前记录当前状态快照
  • checkResponseForCacheBreak() — API 响应后对比快照，检测缓存命中率异常下降
  • notifyCompaction() — 压缩后通知检测器，预期缓存读取会下降（避免误报）
  • notifyCacheDeletion() — 缓存编辑后通知检测器

会话稳定性设计

状态锁存（State Latching）：某些决策（Beta 头部、缓存 TTL、推理努力等级）一旦在会话中确定，便在整个会话期间保持不变。这防止了中途参数抖动导致反复缓存失效。

6. 工具结果预算控制

在查询管线最早期阶段执行，对每条消息中过大的工具结果进行截断。

applyToolResultBudget()

  • 在微压缩之前运行
  • 缓存微压缩仅按 tool_use_id 操作（不检查内容），因此截断对其不可见，两者可组合使用
  • 截断记录可持久化到磁盘（用于 Agent 恢复和 /resume 场景）
  • 无 maxResultSizeChars 限制的工具类型会被排除在外

7. 历史裁剪（History Snip）

特性开关: HISTORY_SNIP

snipCompactIfNeeded()

与微压缩不互斥，两者可同时运行。裁剪移除中间轮次的旧消息，释放的 token 数会传递给自动压缩的阈值判断。

  • 返回 snipTokensFreed — 裁剪释放的估算 token 数
  • 如有变更，产出一个 boundaryMessage
  • tokenCountWithEstimation() 无法感知裁剪的效果（因为 usage 来自受保护的尾部 assistant），所以需要单独传递释放量

8. 微压缩（Micro Compact）

src/services/compact/microCompact.ts · src/services/compact/cachedMicrocompact.ts

在每次 API 调用前执行，是最轻量的上下文回收手段。

时间触发微压缩

src/services/compact/timeBasedMCConfig.ts

参数默认值说明
enabledtrue总开关
gapThresholdMinutes60触发时间间隔（分钟）
keepRecent5保留最近 N 个结果

原理：服务端 Prompt 缓存在 1 小时后过期，此时全部前缀会被重写。既然缓存已失效，不如趁机清除旧工具结果，减小重写体积。

缓存编辑微压缩

特性开关: CACHED_MICROCOMPACT

参数默认值说明
triggerThreshold12触发所需工具结果数
keepRecent3保留最近的结果数
supportedModelsopus-4-6, sonnet-4-6支持的模型列表

原理：利用 API 的 cache_edits 功能在服务端删除工具结果，不修改本地消息，不破坏已缓存的前缀。

状态管理

CachedMCState {
  pinnedEdits: PinnedCacheEdits[]  // 已固定的缓存编辑
  registeredTools: Set<string>     // 已注册的工具结果 ID
  toolOrder: string[]              // 工具结果顺序（用于删除）
  deletedRefs: Set<string>         // 已删除的引用 ID
}

可压缩工具列表

FileRead Bash Grep Glob WebSearch WebFetch FileEdit FileWrite

Token 估算

estimateMessageTokens() 对消息进行粗略 token 估算，带有 4/3 的保守系数。对不同内容块分别计算：文本、thinking、tool_use（名称+输入）、tool_result（文本/图片/文档）。

9. 上下文折叠（Context Collapse）

src/services/contextCollapse/index.ts

特性开关: CONTEXT_COLLAPSE 实验性功能，当前默认禁用

设计理念

提供比消息裁剪更细粒度的上下文管理。通过维护一个"已提交日志"（committed log），将旧的消息组摘要化而不是完全删除。

阈值设计

阈值占有效窗口比例触发动作
提交阈值~90%开始将旧轮次折叠为摘要
阻塞产出阈值~95%阻止新的子代理产出

与自动压缩的关系

当折叠功能启用时，主动自动压缩被禁用（因为自动压缩在 ~93% 触发，恰好位于折叠的两个阈值之间，两者会竞争）。但响应式压缩作为 413 错误的兜底仍然保留。

接口

initContextCollapse()           // 初始化
resetContextCollapse()          // 重置（压缩后调用）
applyCollapsesIfNeeded()        // 应用折叠到消息视图
recoverFromOverflow()           // 溢出恢复
projectView(messages)           // 投射折叠后的上下文视图
注意：折叠是读取时的投射（projection），不修改 REPL 的完整消息历史。摘要存储在折叠存储中，而非 REPL 数组中。这使得折叠可以跨轮次持久化。
✎

10. 完整压缩（Full Compact）

src/services/compact/compact.ts · src/services/compact/prompt.ts

压缩提示词结构

压缩时使用特制的提示词指导模型生成摘要。有两种模板：

全量压缩 BASE_COMPACT_PROMPT

对整个对话进行摘要

部分压缩 PARTIAL_COMPACT_PROMPT

仅摘要最近部分，早期消息保持原样

摘要包含的 9 个章节

  1. 主要需求和意图 — 用户的所有显式请求
  2. 关键技术概念 — 涉及的技术栈和框架
  3. 文件和代码段 — 查看/修改/创建的文件，含代码片段
  4. 错误与修复 — 遇到的错误及解决方案
  5. 问题解决 — 已解决的问题和进行中的排查
  6. 所有用户消息 — 非工具结果的用户发言（完整列出）
  7. 待办任务 — 明确要求但尚未完成的任务
  8. 当前工作 — 压缩前正在进行的工作（含文件名和代码）
  9. 可选下一步 — 下一步计划（需与用户最近请求直接相关）
防止工具调用：压缩提示词以 NO_TOOLS_PREAMBLE 开头和 NO_TOOLS_TRAILER 结尾，强调禁止调用工具。在 Sonnet 4.6 上曾出现模型无视指令尝试工具调用的情况（2.79% vs 4.5 的 0.01%），前置强调可避免浪费轮次。

压缩后处理

src/services/compact/postCompactCleanup.ts

runPostCompactCleanup() 清理压缩后失效的缓存和跟踪状态：

清理的内容

  • 微压缩状态
  • 上下文折叠存储（仅主线程）
  • 记忆文件缓存
  • 系统提示词段缓存
  • 分类器审批记录
  • 推测性检查结果
  • Beta 追踪状态
  • 会话消息缓存

保留的内容

  • 技能内容 — 必须跨多次压缩存活，以便后续压缩附件中包含完整技能文本

恢复的内容

  • CLAUDE.md 文件（通过 session start hooks）
  • 技能文件（每个最大 5KB，总预算 25K）
  • 最多恢复 5 个文件，总预算 50K token
子代理安全：子代理与主线程共享进程和模块级状态。只有主线程的压缩才会重置上下文折叠存储和记忆文件缓存等共享状态，防止子代理压缩破坏主线程状态。

11. 错误恢复机制

src/query.ts 错误处理分支

上下文溢出恢复

错误类型恢复策略
prompt_too_long (413)
  1. 如果上下文折叠启用 → 尝试 recoverFromOverflow()
  2. 如果响应式压缩启用 → 尝试响应式压缩
  3. 否则 → 返回错误消息给用户
max_output_tokens
  1. 首次：将 maxOutputTokens 升级到 ESCALATED_MAX_TOKENS (64K) 重试
  2. 再次失败：返回截断结果
媒体处理错误 移除导致问题的图片/PDF，用文本替代后重试

阻塞限制

在实际调用 API 之前，查询管线会检查是否已达到阻塞限制（effectiveWindow - 3K，约 98%）：

  • 以下场景跳过阻塞检查：compact、session_memory 来源；响应式压缩或上下文折叠管理上下文时
  • 如果达到阻塞限制，直接向用户返回 PROMPT_TOO_LONG_ERROR_MESSAGE，不发送 API 请求

13. 输出 Token 管理

槽位预留优化

默认 max_output_tokens 为 32K-64K，但 p99 实际输出仅 4,911 token，导致 8-16 倍的槽位过度预留。

  • 优化方案：CAPPED_DEFAULT_MAX_TOKENS = 8,000 作为初始上限
  • 不到 1% 的请求会触发上限，这些请求以 64K 重试一次
  • 重试通过 query.ts 中的 max_output_tokens_escalate 逻辑实现

任务预算跟踪

通过 taskBudget 参数传递总预算和剩余量。压缩时，根据 finalContextTokensFromLastResponse()（input + output，不含 cache）计算消耗，从剩余预算中扣除。

上下文构成公式

总 Token 数 =
  系统提示词 Token（静态前缀 + 动态后缀）
  + 用户上下文 Token（CLAUDE.md、记忆文件等）
  + 系统上下文 Token（Git 状态、日期等）
  + 工具定义 Token（所有已注册工具的 JSON Schema）
  + 消息 Token（用户消息 + 助手消息 + 附件）
  + 输出 Token（模型生成的回复）

其中消息 Token = Σ(
  文本块 + thinking 块 + tool_use 块(名称+输入)
  + tool_result 块(文本/图片/文档)
  + 附件内容
)
