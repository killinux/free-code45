架构总览

内存管理系统由 5 个层次 协同工作，从实时计算到持久化存储层层递进：

层次职责开销持久性
Token 计量 度量上下文窗口使用率 零（仅计算） 每次请求
微压缩 轻量级工具结果清理 零 / cache_edits API 会话内
自动压缩 完整对话摘要化 1 次 API 调用（传统路径） 会话内
会话记忆 结构化会话笔记，用于快速压缩 分叉子代理（共享缓存） 会话文件
持久记忆 跨会话的持久化知识 分叉子代理（共享缓存） 永久文件

1. Token 计量层

src/utils/tokens.ts

核心函数：tokenCountWithEstimation()

这是度量上下文大小的标准函数。所有阈值判断（自动压缩、会话记忆等）都使用它。

算法流程

  1. 从消息列表末尾向前查找最后一条包含有效 usage 数据的 assistant 消息
  2. 计算完整上下文大小：input_tokens + cache_creation + cache_read + output_tokens
  3. 回溯到具有相同 message.id 的第一条同源记录（处理并行工具调用场景）
  4. 对锚点之后的新增消息进行粗略估算并累加
并行工具调用处理：当模型发起多个工具调用时，流式输出会将同一个 message.id 拆分成多条 assistant 记录。该函数会回溯到第一条同源记录，确保夹在中间的 tool_result 不被遗漏。

相关函数

函数返回值用途
getTokenCountFromUsage() 从单个 Usage 对象计算完整上下文大小 其他函数的基础构建块
tokenCountFromLastAPIResponse() 最后一次 API 调用的完整上下文大小 压缩前的基线测量
finalContextTokensFromLastResponse() input + output（不含缓存） 任务预算剩余量计算
messageTokenCountFromLastAPIResponse() 仅输出 token 数 度量单次回复大小（不可用于阈值判断）

2. 微压缩（Micro Compact）

src/services/compact/microCompact.ts

最轻量的上下文回收方式，在每次 API 调用前执行。

路径 A：基于时间触发

直接修改消息

当距上一条 assistant 消息的时间间隔超过阈值时，服务端缓存已经过期，因此直接清除旧的工具结果内容。

  • 将内容替换为 [Old tool result content cleared]
  • 保留最近 N 个工具结果（最少保留 1 个）
  • 重置缓存微压缩状态（因为服务端缓存已失效）

路径 B：缓存编辑微压缩

Cache Edits API

当缓存仍然有效时，通过 cache_edits API 在服务端删除工具结果，不破坏已缓存的前缀。

  • 不修改本地消息内容
  • 触发/保留阈值由 GrowthBook 远程配置控制
  • 通过 CachedMCState 跟踪工具结果
  • 将 pendingCacheEdits 排队等待 API 层处理

可压缩的工具类型

仅以下工具类型的结果可以被微压缩：

FileRead Bash Grep Glob WebSearch WebFetch FileEdit FileWrite

3. 自动压缩（Auto Compact）

src/services/compact/autoCompact.ts

阈值计算

有效窗口 = min(模型窗口大小, CLAUDE_CODE_AUTO_COMPACT_WINDOW) - 20K（输出预留）
自动压缩阈值 = 有效窗口 - 13K   // 约 93% 占用率时触发
阻塞限制 = 有效窗口 - 3K       // 约 98%，阻塞用户输入
13K 自动压缩缓冲 Token 数
20K 警告/错误阈值缓冲
3K 手动压缩缓冲（阻塞限制）
20K 摘要最大输出 Token 数

执行优先级

  1. 会话记忆压缩（零 API 开销）— 使用已提取的会话笔记作为摘要
  2. 传统压缩（1 次 API 调用）— 将完整对话发送给 LLM 生成摘要

压缩后清理

  • 重置 lastSummarizedMessageId（旧的 UUID 在压缩后不再存在）
  • runPostCompactCleanup() 恢复技能和 CLAUDE.md 文件（每个技能 5KB，总预算 25K）
  • 通知 prompt 缓存中断检测器，预期缓存读取量会下降

熔断器机制

最大连续失败次数 = 3
连续失败 3 次后，自动压缩在本次会话的剩余时间内停止重试。 背景：曾有 1,279 个会话连续失败 50 次以上（最多达 3,272 次），全球每天浪费约 25 万次 API 调用。

4. 会话记忆（Session Memory）

src/services/SessionMemory/sessionMemory.ts

维护一个结构化的 Markdown 文件，记录当前对话的关键信息。通过采样后钩子（post-sampling hook）定期运行。

触发阈值

参数默认值说明
minimumMessageTokensToInit 10,000 首次触发提取所需的最低 token 数
minimumTokensBetweenUpdate 5,000 两次提取之间所需的 token 增量
toolCallsBetweenUpdates 3 两次提取之间所需的最少工具调用次数

触发逻辑

是否提取 = (满足 token 阈值 AND 满足工具调用阈值)
         OR (满足 token 阈值 AND 最后一轮没有工具调用)

Token 阈值是必要条件 — 即使工具调用次数达标，也必须等到 token 增长满足要求才会触发提取。

执行机制

  1. 检查特性开关 tengu_session_memory（缓存值，非阻塞）
  2. 创建隔离的 subagentContext，避免污染父级缓存
  3. 在 ~/.claude/session-memory/MEMORY.md 设置会话记忆文件
  4. 通过 FileReadTool 读取当前记忆内容
  5. 使用 buildSessionMemoryUpdatePrompt() 构建提取提示词
  6. 通过 runForkedAgent() 运行分叉子代理，querySource: 'session_memory'
  7. 子代理仅允许使用 FileEdit 编辑指定的记忆文件

记忆文件结构（12 个分区）

会话标题、当前状态、任务规格、文件/函数、工作流、错误与修正、系统文档、经验总结、关键成果、工作日志等。

  • 每个分区上限 2K token
  • 总计 12K token 限制
  • 在用于压缩前由 truncateSessionMemoryForCompact() 截断超长分区

5. 会话记忆压缩（SM Compact）

src/services/compact/sessionMemoryCompact.ts

使用已有的会话记忆作为摘要，替代调用 API 生成摘要。零 API 开销。

消息保留配置

参数默认值说明
minTokens10,000压缩后最少保留的 token 数
minTextBlockMessages5最少保留的包含文本内容的消息数
maxTokens40,000保留 token 数的硬上限

calculateMessagesToKeepIndex() 算法

  1. 从 lastSummarizedMessageId + 1 开始
  2. 向前扩展，直到同时满足两个最低保留量（token 数和文本消息数）
  3. 达到 maxTokens 硬上限时停止
  4. 不越过最后一个压缩边界标记
  5. 调用 adjustIndexToPreserveAPIInvariants() 调整索引

adjustIndexToPreserveAPIInvariants() 的作用

确保 tool_use/tool_result 配对不被拆分，且共享同一 message.id 的思考块（thinking blocks）会被一起保留。

该函数修复的 Bug 场景：流式输出将同一 message.id 的内容拆分为多条记录。如果切割边界恰好落在拆分记录之间，normalizeMessagesForAPI 合并时会产生孤立的 tool_result 引用，导致 API 报错。

两种场景

正常情况

lastSummarizedMessageId 已设置。仅保留该 ID 之后的消息，并向前扩展以满足最低保留量。

恢复会话

lastSummarizedMessageId 未设置，但会话记忆已有内容。保留所有消息，使用会话记忆作为摘要。

6. 持久记忆（Auto Memory）

src/services/extractMemories/extractMemories.ts

从当前会话中提取持久化记忆，写入 ~/.claude/projects/<path>/memory/ 目录。在每轮完整查询循环结束时运行。

触发条件

  • 最后一轮 assistant 回复不包含工具调用（自然对话断点）
  • 通过 handleStopHooks() 调用
  • 仅对主代理生效（子代理不触发）
  • 节流控制：每 N 个合格轮次执行一次（tengu_bramble_lintel，默认 1）

执行设计

互斥机制

如果主代理本轮已经直接写入了 memory 文件，后台提取会被跳过，同时游标前移跳过该范围。

重叠保护

如果在提取进行中有新的请求到来，上下文会被暂存。当前提取完成后，会用最新的暂存上下文执行一次尾随提取。

资源限制

  • maxTurns: 5 — 硬限制，防止子代理陷入验证循环
  • 排空超时: 60 秒 — drainPendingExtraction() 在优雅关闭前等待后台提取完成
  • skipTranscript: true — 避免与主线程产生竞态条件

权限模型

createAutoMemCanUseTool() 定义了严格的沙箱环境：

工具权限
FileRead、Grep、Glob不限制（只读操作）
Bash仅限只读命令（ls、find、grep、cat、stat、wc、head、tail）
FileEdit、FileWrite仅限 memory 目录
REPL允许（内部原语仍受上述规则限制）
其他所有工具拒绝

用户消息 | v +-------------------------+ | 微压缩消息 | <-- API 调用前执行 | (基于时间触发 | | 或缓存编辑) | +-------------------------+ | v +-------------------------+ | API 采样 | <-- 发送给 Claude | (如有待处理的 | | cache_edits 一并发送) | +-------------------------+ | v +-------------------------+ | 采样后钩子: | <-- 每次 API 响应后 | 会话记忆提取 | | (满足阈值时触发) | +-------------------------+ | v +-------------------------+ | 是否需要自动压缩? | <-- Token 使用量 > ~93% | | | 是: | | 1. 尝试会话记忆压缩 | (零 API 开销，使用会话笔记) | 2. 尝试传统压缩 | (1 次 API 调用，完整摘要) | | | 否: 继续 | +-------------------------+ | v +-------------------------+ | 工具执行循环 | <-- 执行工具调用 +-------------------------+ | v (无更多工具调用时) +-------------------------+ | 停止钩子: | <-- 本轮结束 | 提取持久记忆 | | (后台分叉子代理) | | | | 写入路径: | | ~/.claude/projects | | /<path>/memory/ | +-------------------------+

8. 安全防护机制

递归防护

以下 querySource 值会被禁止触发自动压缩：

  • session_memory — 会导致死锁（分叉子代理）
  • compact — 会导致死锁（分叉子代理）
  • marble_origami — 上下文折叠代理，存在模块级状态冲突

熔断器

  • 连续 3 次自动压缩失败后，停止重试
  • 防止上下文不可恢复地超限时反复调用 API
  • 成功时重置计数

API 不变量保护

  • tool_use / tool_result 配对永远不会被拆分
  • 共享同一 message.id 的思考块始终一起保留
  • 从 messagesToKeep 中过滤旧的压缩边界标记，防止二次裁剪

互斥控制

  • 会话记忆提取使用 sequential() 串行化包装
  • 如果主代理本轮已写入记忆，则跳过后台提取
  • 重叠时：暂存上下文，仅执行一次尾随提取
  • 上下文折叠和响应式压缩模式下禁用主动自动压缩
