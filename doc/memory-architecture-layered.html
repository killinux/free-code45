内存管理架构 — 原理分层（修订版）

基于代码模块边界和真实依赖关系重新推导的 6 层架构
每一层的存在都有代码级的依赖证据

前一版的问题

前一版 8 层架构有几个不诚实之处：

  • L1 (物理存储) 和 L3 (访问控制) 被人为拆分 — 代码中它们在同一个模块 memdir/ 里，paths.ts 和 teamMemPaths.ts 同时处理路径和安全。拆成两层是概念上的"好看"而非代码上的"真实"。
  • L7 (调度集成) 不是一个层 — 钩子和子代理调度不构成独立的抽象层，它们是 query.ts 和各模块内部的胶水代码。给胶水代码命名为"层"是抬高了它的地位。
  • L8 (控制策略) 不是一个层 — 特性开关、熔断器、配置渗透在每一层中。它是横切关注点（cross-cutting concern），不是一个分层。强行放在顶部暗示它"管理其他层"，但实际上它只是 if 判断散落在各处。
  • L5 (提取生成) 混合了三个独立子系统 — Session Memory、Auto Memory、Auto Dream 有不同的触发时机、不同的输出目标、不同的生命周期。把它们塞进一层掩盖了差异。

修订原则：每一层的边界必须能在代码的 import 依赖关系中找到证据。如果两个东西互相导入、共享生命周期，它们是同一层；如果一个东西只被另一个东西导入，它是后者的子模块而非独立的层。

L1

存储与寻址

回答的问题：记忆在哪？谁能碰？
存在理由：所有上层模块都 import paths.ts 或 teamMemPaths.ts 来确定读写位置。这是代码中被依赖最多的模块，改它会影响全部上层。
src/memdir/paths.ts · teamMemPaths.ts

路径解析（3 级优先级）

  • 1. 环境变量覆盖 — CLAUDE_COWORK_MEMORY_PATH_OVERRIDE
  • 2. settings.json — autoMemoryDirectory（仅信任来源）
  • 3. 默认约定 — ~/.claude/projects/<git-root>/memory/
  • Worktree 共享：findCanonicalGitRoot()
  • 记忆化：getAutoMemPath() 缓存结果避免 stat

路径安全（与存储合并的原因）

  • 在代码中 teamMemPaths.ts 同时做目录构造和安全验证
  • 5 种遍历攻击防护（null/URL 编码/Unicode/反斜杠/绝对路径）
  • realpathDeepestExisting() 符号链接解析
  • 前缀攻击防护（要求分隔符紧跟）
  • 文件 0o600 / 目录 0o700
设计原理

路径即安全边界。前一版把"存储"和"访问控制"拆成两层是错误的。代码中 paths.ts 的每个函数既在构造路径、又在验证路径安全。isAutoMemPath() 同时回答"这个路径在记忆目录下吗"和"这个路径安全吗"。在文件系统方案中，路径和安全是同一个关注点，不可分割。

L2

数据契约

回答的问题：记忆长什么样？读写双方遵守什么格式？
存在理由：memoryTypes.ts 定义的类型和 frontmatter 格式被 L3 (扫描时解析)、L5 (提取时写入) 和 L4 (注入时展示) 共同依赖。它是读写双方的契约，改它需要同步修改两侧。
src/memdir/memoryTypes.ts · SessionMemory/prompts.ts (模板部分)

持久记忆格式

  • 4 种类型：user / feedback / project / reference
  • Frontmatter 字段：name / description / type
  • 内容约定：feedback/project 带 **Why:** 和 **How to apply:**
  • MEMORY.md 条目：一行 ~150 字符 - [Title](file.md) — 钩子

会话记忆格式

  • 12 个分区：Title / State / Task / Files / Workflow / Errors / Docs / Learnings / Results / Worklog 等
  • 每分区上限：2,000 token
  • 总计上限：12,000 token
  • 分区标题和斜体描述不可修改，仅编辑内容
设计原理

契约层独立于实现。L2 不做任何 I/O，不调用 LLM，不写文件。它只回答"记忆应该长什么样"。这让 L3 (扫描) 和 L5 (提取) 可以独立演化 — 只要双方遵守相同的 frontmatter 格式和分区模板，它们不需要知道彼此的存在。

L3

扫描与索引

回答的问题：当前有哪些记忆？它们的元信息是什么？
存在理由：memoryScan.ts 被 L4 (findRelevantMemories) 和 L5 (extractMemories 的清单注入) 共同依赖。它把文件系统状态转化为内存中的结构化清单，是读写两侧的共享基础设施。
memoryScan.ts · memdir.ts · memoryAge.ts

文件扫描 memoryScan.ts

  • 递归扫描 .md 文件，排除 MEMORY.md
  • Promise.allSettled() 并行读 frontmatter
  • 仅读前 30 行（性能优化）
  • 单次 I/O：stat + 读取同时完成
  • 按 mtime 降序，返回前 200 个 MemoryHeader

索引管理 memdir.ts

  • MEMORY.md 加载与截断（200 行 / 25KB）
  • 先按行截断，再按字节截断
  • 追加截断警告（指明触发了哪个限制）
  • ensureMemoryDirExists() 幂等初始化

新鲜度标记 memoryAge.ts

  • memoryAge() → 'today' / 'yesterday' / 'N days ago'
  • ≤1 天：无警告
  • >1 天：附加 <system-reminder> 提醒模型验证

为什么是独立一层

  • L4 (检索) 调用 scanMemoryFiles() 获取清单
  • L5 (提取) 调用 formatMemoryManifest() 注入提示词
  • L6 (压缩桥) 通过 memdir.ts 读取 MEMORY.md
  • 被三个上层共同依赖，证明它是真实的分界面
设计原理

扫描是记忆系统的"感知器官"。它把文件系统的物理状态（散落的 .md 文件）转化为上层可消费的结构化信息（MemoryHeader 列表 + MEMORY.md 内容）。所有上层操作 — 检索、提取、压缩 — 都不直接触碰文件系统，而是通过 L3 提供的抽象来感知记忆世界的状态。

L4

检索与注入

回答的问题：当前对话需要哪些记忆？怎么放进去？
存在理由：findRelevantMemories.ts 是唯一调用外部 LLM (Sonnet) 做相关性判断的模块。它依赖 L3 (扫描) 提供清单，被 query.ts (通过 attachments.ts) 消费。它有独立的 I/O 模式（LLM 调用）和独立的预取生命周期（Disposable），与其他层明显不同。
findRelevantMemories.ts · attachments.ts (记忆部分)

相关性选择算法

  • 输入：L3 的 MemoryHeader 清单
  • 过滤：已展示过的 + 已读过的
  • 调用 Sonnet 3.5 判断相关性
  • 输出：最多 5 个文件名 (JSON Schema 约束)
  • 工具过滤：使用中的工具文档不展示，但警告仍展示

注入机制

  • 作为 relevant_memories 附件注入
  • 每文件 4KB，每轮 5 个，会话 60KB
  • 截断时追加 FileReadTool 链接
  • 陈旧记忆附加新鲜度警告头
  • 压缩后重置累计计数（可重新注入）

非阻塞预取

  • startRelevantMemoryPrefetch() 在 query.ts 入口启动
  • 使用 using 语法（Disposable），所有退出路径自动释放
  • 消费点在 1600 行左右检查 settledAt
  • 永不阻塞：未就绪就跳过

为什么 LLM 检索是独立层

  • 它是唯一在读路径上调用外部 LLM 的模块
  • 有独立的异步生命周期（预取 + Disposable）
  • 有独立的失败模式（Sonnet 调用失败 → 跳过注入）
  • 与 L5 (写入) 完全独立 — 代码中没有交叉导入
设计原理

读写分离 + 读路径永不阻塞。L4 (检索) 和 L5 (提取) 在代码中没有任何 import 关系，它们通过 L3 的共享清单间接协作。预取的 Disposable 模式保证即使 Sonnet 调用超时或失败，主对话也不受影响。这是整个系统最关键的延迟保证。

L5

三个写入者（并列而非嵌套）

回答的问题：什么信息值得记住？由谁在什么时候写入？
存在理由：三个子系统触发时机不同、输出目标不同、生命周期不同，但共享 runForkedAgent() 基础设施和 createAutoMemCanUseTool() 权限沙箱。它们是同一层的三个并列实现，不是一个系统的三个阶段。
SessionMemory/ · extractMemories/ · autoDream/

L5a 会话记忆

  • 触发：postSamplingHook
  • 条件：token ≥10K (初始) + 增量 5K + 工具调用 3
  • 执行者：forkedAgent
  • 权限：仅 Edit 单个文件
  • 输出：session-memory/MEMORY.md
  • 生命：会话级

L5b 持久记忆

  • 触发：handleStopHooks
  • 条件：轮次结束 + 无工具调用
  • 执行者：forkedAgent
  • 权限：Read/Grep/Glob + 只读 Bash + Edit/Write 限 memory/
  • 输出：projects/<path>/memory/*.md
  • 生命：永久

L5c 记忆合并

  • 触发：后台（非钩子）
  • 条件：时间 ≥24h + 会话 ≥5 + 锁可用
  • 执行者：forkedAgent
  • 权限：同 L5b
  • 输出：更新已有 memory/*.md
  • 生命：跨会话

三者共享的基础设施

  • runForkedAgent() — 创建隔离子代理，共享父级 prompt 缓存
  • createAutoMemCanUseTool() — 权限沙箱（autoDream 直接复用 extractMemories 的导出）
  • queryClass 标记 — 防止子代理触发自动压缩（session_memory / extract_memories）
  • 互斥逻辑 — 主代理写入时跳过 L5b；L5b 运行中暂存后续请求
设计原理

三个独立写入者，不是一个管线的三个阶段。前一版的错误是暗示它们是串行管线。实际上：Session Memory 在每次 API 响应后触发，Auto Memory 在轮次结束时触发，Auto Dream 在跨会话空闲时触发。它们时间尺度不同（秒级 / 分钟级 / 天级），互不等待，互不依赖（除了 L5c 复用了 L5b 的权限沙箱函数）。

L6

上下文桥接

回答的问题：记忆系统如何与上下文压缩系统交互？
存在理由：sessionMemoryCompact.ts 同时 import 了 SessionMemory/ 的状态和 compact/ 的数据结构，是唯一横跨记忆系统和上下文管理的模块。它不属于记忆系统（不写入记忆），也不完全属于压缩系统（依赖记忆内容），而是两个系统的桥梁。
sessionMemoryCompact.ts · postCompactCleanup.ts

SM Compact（记忆 → 压缩）

  • 读取 L5a 的会话记忆内容作为摘要
  • 零 API 调用 — 不需要 LLM 生成摘要
  • 保留 10K-40K token 的近期消息
  • 最少 5 条文本消息
  • adjustIndexToPreserveAPIInvariants() 保护配对
  • 优先于传统压缩（成本为零）

Post-Compact Cleanup（压缩 → 记忆）

  • 重置 lastSummarizedMessageId
  • 清空 memoryFiles 缓存（重新加载 CLAUDE.md）
  • 清空 systemPromptSections（重建系统提示词）
  • 通过 sessionStartHook 恢复 CLAUDE.md 附件
  • 保留技能内容（跨压缩存活需要）
  • 仅主线程压缩重置共享状态
设计原理

桥接层是两个独立系统的唯一耦合点。记忆系统 (L1-L5) 和上下文压缩系统 (compact/) 是两个独立关注点。L6 是它们之间的"胶水"：SM Compact 把记忆的产出喂给压缩器，Post-Compact Cleanup 在压缩后重新初始化记忆加载。没有 L6，两个系统可以完全独立运行 — 但记忆就无法帮助压缩降成本，压缩也不会恢复记忆。L6 的存在让"会话记忆写了什么"直接变成"压缩的摘要是什么"，节省一次 API 调用。

横切关注点（不是层）

以下内容渗透在多层中，不构成独立的架构层：

  • 特性开关 — getFeatureValue_CACHED_MAY_BE_STALE() 散布在 L4 / L5a / L5b / L5c / L6 的入口处。它不是"管理其他层"的顶层，而是每个模块自己的 if 判断。
  • 环境变量覆盖 — CLAUDE_CODE_DISABLE_AUTO_MEMORY 等在各模块的初始化函数中检查。
  • 熔断器 — 仅存在于 autoCompact (L6) 中，不是全局机制。
  • 权限沙箱 — createAutoMemCanUseTool() 在 L5b 中定义，被 L5b 和 L5c 共享，本质是 L5 的内部基础设施。
  • 钩子调度 — postSamplingHook (触发 L5a) 和 handleStopHooks (触发 L5b) 是 query.ts 的职责，不是记忆系统自己的层。

前一版的错误是把这些横切关注点强行塞成"层"，暗示它们有独立的模块和接口。实际上它们没有。

修订后的跨层数据流

用户输入 (query.ts) | +------> [L4] startRelevantMemoryPrefetch() <非阻塞异步> | | | [L3] scanMemoryFiles() | | | [L1] 读取文件系统 (paths.ts 确定目录) | | | [L2] 解析 frontmatter (memoryTypes.ts 验证) | | | [L4] 调 Sonnet 3.5 选 Top-5 <可能跳过> | | | [L4] 注入为 attachment (4KB/文件, 5 个/轮) | v API 调用 (Claude) <-- 注入的记忆在上下文中 | +------> [L6] 上下文使用率 > 93%? | | | YES: SM Compact ← 读取 [L5a] 的会话记忆 | | | 或: Legacy Compact (1 次 API 调用) | | | Post-Compact: 重置记忆缓存, 恢复 CLAUDE.md | +------> [L5a] postSamplingHook | | | token增量 ≥ 5K 且 (工具调用 ≥ 3 或 无工具调用)? | | | YES: forkedAgent → [L2] 按 12 分区模板写入 | → [L1] session-memory/MEMORY.md | v 轮次结束 (无工具调用) | +------> [L5b] handleStopHooks | | | [L3] formatMemoryManifest() 注入提示词 | | | forkedAgent → [L2] 按 4 类型写入 | → [L1] projects/<path>/memory/*.md | +------> [L5c] 24h + 5 会话? <独立后台> | YES: 合并去重 → [L1] 更新已有 .md
★

为什么是 6 层而不是 8 层

合并的 2 层

  • 旧 L1+L3 → 新 L1
    存储和安全在代码中不可分割（paths.ts 同时做两件事）
  • 旧 L7 (调度) 删除
    钩子调度是 query.ts 的职责，不是记忆系统的层
  • 旧 L8 (控制) 降级为横切
    特性开关是散布的 if 判断，没有独立的模块和接口

新增的结构性认知

  • L5 明确为并列三写入者
    不再暗示串行管线，每个有独立触发和输出
  • L6 明确为桥接层
    前一版的"生命周期层"混入了压缩逻辑，但压缩不属于记忆系统
  • 读写分离由 L3 (扫描) 保证
    L4 (读) 和 L5 (写) 没有直接依赖，通过 L3 共享清单
层次验证标准

每一层的存在都通过了以下测试：
1. 依赖证据 — 至少被两个不同的上层模块 import
2. 变更隔离 — 修改本层不需要同时修改上层
3. 替换可行 — 理论上可以用不同的实现替换（如 L1 从文件换成 SQLite，L4 从 Sonnet 换成向量检索）
4. 关注点单一 — 用一句话能说清楚本层回答什么问题
