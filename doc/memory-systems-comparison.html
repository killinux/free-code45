1. Mem0 — LLM-as-Memory-Manager

核心创新：写时 LLM 决策

Mem0 的最独特设计是把 LLM 当作记忆管理器。每次新事实写入时，系统检索 Top-10 语义相似的现有记忆，然后让 LLM 通过工具调用决定执行哪个操作：

操作触发条件行为
ADD无语义等价的现有记忆生成新 ID，插入新记忆
UPDATE现有记忆可补充信息复用原 ID，附带 old_memory 历史
DELETE检测到与现有记忆矛盾删除被矛盾的记忆
NOOP事实重复或无关不变
关键洞察：冲突解决和去重发生在写入时，通过 LLM 推理完成；而不是在读取时。这意味着记忆库始终保持"清洁"状态，检索时不需要再次去重。

两种变体

Mem0 (基础版)

  • 纯向量库（Qdrant 默认）
  • 每条记忆是自然语言事实 + 嵌入向量
  • 检索：稠密向量 Top-K
  • 简单、快速、生产就绪

Mem0^g (图谱版)

  • Neo4j 知识图谱
  • 两阶段提取：实体抽取 + 关系生成
  • 有向标记图：节点（人/地点）+ 边（lives_in/prefers）
  • 双重检索：实体导航 + 三元组嵌入匹配

LOCOMO 基准成绩

问题类型Mem0Mem0^g
单跳 (Single-Hop)67.1365.71
多跳 (Multi-Hop)51.1547.19
开放域 (Open-Domain)72.9375.71

• 比 OpenAI Memory 准确率高 26%
• p95 延迟降低 91%（1.44s vs 17.12s 全上下文）
• 每次对话约 7K token（vs 全上下文 26K，vs Zep 基线 600K）
• 时序推理改进 50%

2. Letta / MemGPT — LLM 操作系统

核心隐喻：LLM 即操作系统

MemGPT 论文提出"LLM 作为操作系统"的范式：

  • LLM 是 CPU，处理上下文中的指令
  • 上下文窗口是 RAM，容量有限但访问快
  • 外部存储是磁盘，容量无限但需要"换页"加载
  • 代理本身是进程管理器，通过工具调用主动管理自己的内存

两层记忆架构

一层：主上下文（Main Context / "RAM"）

  1. 系统指令 — 只读，包含代理逻辑和工具定义
  2. 工作上下文 / 核心记忆（Core Memory）
    • 可编辑的文本块（通常包含 persona 和 human）
    • 始终在提示词中可见
    • 每块约 2KB，由代理自己维护
  3. FIFO 消息队列 — 最近对话历史，满时被淘汰或摘要

二层：外部上下文（External Context / "磁盘"）

  1. Recall Memory — 完整对话日志，可按关键词/时间搜索
  2. Archival Memory — 无界向量索引知识库，长期事实存储

代理自编辑工具

代理通过以下函数调用主动编辑自己的记忆：

工具作用
core_memory_append(section, content)向核心块追加内容
core_memory_replace(section, old, new)编辑核心块
archival_memory_insert(content)写入向量库
archival_memory_search(query, page)语义搜索（支持分页）
conversation_search(query)搜索 recall memory

request_heartbeat=true 标志使工具调用返回后立即重新调用 LLM，实现链式记忆操作（搜索 → 读取 → 追加到核心）。

内存压力的优雅处理

75% 警告

当上下文使用率达到 ~75% 时，系统注入一条系统消息警告代理"上下文正在填满"，提示代理将重要细节持久化到归档记忆。这是自主压缩 — 由代理决定保留什么。

100% 冲刷

当上下文溢出时，队列管理器淘汰最旧的 FIFO 消息：

  • 被淘汰的内容被递归摘要化
  • 摘要插入到工作上下文
  • 完整消息仍可通过 recall memory 访问
关键洞察：这把"上下文溢出"从硬错误变成了优雅的分页事件。代理可以无限期运行，因为永远不会"忘记"任何东西 — 只是某些东西被分页到了"磁盘"。

3. Zep / Graphiti — 双时态知识图谱

三层知识图谱架构

G = (N, E, φ)，包含三个子图：

  1. Episode 子图（G_e） — 原始 episode 节点（消息、文本、JSON）。无损数据存储，每个派生事实都可追溯到其 episode（出处追溯）
  2. 语义实体子图（G_s） — 提取的实体节点 + 语义边（关系）。实体通过嵌入余弦相似度 + 全文匹配去重
  3. 社区子图（G_c） — 强连通实体的高层聚类，通过 LLM 生成摘要，使用 Leiden 风格的标签传播社区检测构建

双时态模型 — 决定性特征

每条边携带 4 个时间戳，跨越两条时间轴：

时间轴 T (事件时间)：
  t_valid       — 事实在现实世界中开始有效的时间
  t_invalid     — 事实在现实世界中失效的时间

时间轴 T' (事务时间)：
  t'_created    — 系统学到该事实的时间
  t'_expired    — 系统将其标记为失效的时间

这是真正的双时态数据库模型（借鉴自时态 DB 文献）。它支持以下查询：

  • "我们在 3 月 5 日认为 X 是什么？" — 使用 t' 时间轴
  • "3 月 5 日实际是什么？" — 使用 t 时间轴
  • "2024 年用户偏好的工具是什么，现在又是什么？" — 跨时间对比

边失效算法（永不删除）

当摄入新边时，系统通过 LLM 推理与语义相似的现有边比较。检测到时间重叠的矛盾时：

  1. 不删除被矛盾的旧边
  2. 在旧边上设置 t_invalid = 新边的 t_valid
  3. 新边正常插入，建立替代关系
设计意图：事实永远不会被删除，只会被失效。这保留了完整历史，支持任意时间点的查询。Mem0 用 DELETE 覆盖矛盾，Zep 用失效保留历史 — 这是两种截然不同的哲学。

三方融合检索

检索管线：φ(Search) → ρ(Reranker) → χ(Constructor)

  1. 余弦语义搜索 — 节点/边嵌入向量
  2. BM25 全文检索 — 通过 Lucene（Neo4j）进行关键词匹配
  3. BFS 图遍历 — 在 n 跳邻域内广度优先搜索

重排器融合三方信号；构造器组装最终上下文块。检索延迟 < 200ms，与图规模无关。

基准成绩

  • DMR (Deep Memory Retrieval)：94.8%（GPT-4-Turbo），超越 MemGPT 的 93.4%
  • LongMemEval：使用 GPT-4o 提升 18.5%
  • 延迟：~2.58s vs 基线 28.9s（降低 90%）
  • 上下文压缩：在企业级聊天语料上从 115K token 压缩到 1.6K token

4. 关键架构对比维度

"智能"位于何处

系统智能位置含义
Claude Code 读取时（检索阶段） LLM 在每次查询时判断哪些记忆相关
Mem0 写入时（更新决策） LLM 在每次写入时决定 ADD/UPDATE/DELETE/NOOP
Letta 代理本身（自编辑） 代理通过工具调用主动管理记忆
Zep 摄入管线（提取阶段） 实体提取 + 边失效都在数据进入图谱时完成

处理矛盾的不同哲学

Claude Code — 后台合并

Auto Dream 在 24h + 5 会话后运行，合并去重。矛盾在合并阶段解决。

Mem0 — 写时覆盖

"最新真理获胜"。LLM 检测矛盾后直接 DELETE 旧记忆。简单粗暴。

Letta — 代理决定

代理自己 core_memory_replace。质量取决于代理的编辑纪律。

Zep — 失效保留

永不删除，标记 t_invalid。完整历史可追溯，支持时间旅行查询。

系统扩展机制瓶颈

5. Claude Code 在四者中的定位

Claude Code 的独特之处

  • 唯一一个文件优先的方案 — 其他三个都是数据库后端
  • 唯一支持 Git 版本控制的方案 — 记忆变更可追溯回滚
  • 唯一与 IDE/CLI 深度集成的方案 — CLAUDE.md 与记忆统一
  • 唯一受生物学启发的合并机制 — Auto Dream 模拟睡眠记忆整理
  • 最复杂的渐进式压缩 — 5 层（微压缩 / SM 压缩 / 自动压缩 / 折叠 / Snip）

Claude Code 的弱点（与其他三者对比）

弱点对比对象差距
检索可扩展性差 vs Mem0 / Zep 200 文件硬上限 vs 百万级向量/图
无时态推理 vs Zep 仅有 mtime + 1 天警告 vs 双时态模型
无矛盾检测 vs Mem0 / Zep 需要 Auto Dream 后台合并 vs 写时即时检测
代理无主动编辑权 vs Letta 子代理被动提取 vs 主代理自由读写
无知识图谱 vs Mem0^g / Zep 记忆扁平化 vs 关系网络
无公开基准 vs 全部 无法量化对比

6. 跨系统综合启发

从 Mem0 借鉴

1. 写时矛盾检测
Claude Code 当前依赖 Auto Dream 在 24h 后才合并矛盾。可以借鉴 Mem0 的 ADD/UPDATE/DELETE/NOOP 模式，在子代理提取阶段即时检测矛盾。例如：当提取出"用户偏好 TypeScript"时，检索是否已有"用户偏好 JavaScript"，如有，让 LLM 判断是补充还是替换。
2. 历史表追踪
Mem0 用单独的 history 表记录所有更新，可审计。Claude Code 已经有 Git 版本控制，但可以增加结构化的 _history.md 文件记录每次记忆变更的原因。

从 Letta 借鉴

3. 代理自编辑权
Letta 让主代理直接管理自己的记忆（core_memory_append 等工具）。Claude Code 的子代理是被动的"记忆员"，主代理无法主动整理记忆。可以给主代理增加 memory_edit 工具，允许它在对话中主动维护记忆，而不依赖后台子代理。
4. 75% 警告机制
Letta 在上下文使用率达 75% 时主动提示代理"该持久化重要信息了"。Claude Code 在 ~93% 才触发自动压缩，错失了让代理自主整理的机会。可以增加 75% 软警告 — 给主代理一个机会用 memory_edit 主动归档关键信息。
5. Heartbeat 链式调用
Letta 的 request_heartbeat=true 让记忆操作可以串成链（搜索 → 读取 → 写入）。Claude Code 的子代理虽然有最多 5 轮，但缺乏这种显式的链式语义。

从 Zep 借鉴

6. 双时态模型
这是 Zep 最有价值的设计。Claude Code 当前只有 mtime（事务时间），没有事件时间。例如："2024 年项目用 React"和"2026 年项目用 Vue"应该都保留，而不是覆盖。可以在 frontmatter 增加： 
---
type: project
valid_from: 2026-01-01
valid_until: null  # 仍然有效
created_at: 2026-01-15
---
7. 永不删除 + 失效标记
当用户的偏好/项目状态变化时，不删除旧记忆，而是标记 valid_until。这样可以回答"用户去年偏好什么"、"这个项目以前用什么技术栈"等历史查询。
8. 三方融合检索
Zep 的 余弦 + BM25 + BFS 三方融合是检索的"瑞士军刀"。Claude Code 可以增加 BM25 全文检索作为 LLM 选择的预筛选层，把候选从 200 缩到 ~30。Auto Memory 目录加索引文件即可，无需引入向量数据库。
9. 出处追溯（Episode 子图）
Zep 的每个事实都可追溯到原始 episode。Claude Code 的提取记忆没有"来源"信息。可以在 frontmatter 增加 source_session: <session-id>，帮助用户验证记忆是否准确。

8. 总结：四种哲学，四种理想场景

场景最佳选择原因
本地 CLI / IDE 集成开发 Claude Code 文件可读可编辑，与代码库一起 Git 管理
消费级聊天机器人 / 个性化助手 Mem0 快速演化的用户偏好，写时去重保持记忆库清洁
持久身份的长寿命单代理 Letta 核心记忆 + 分页 + 自编辑，代理可"持续工作"数月
企业级时序事实推理 Zep 双时态模型支持"何时知道何事"的复杂查询
核心洞察：没有"最好"的记忆系统，只有"最适合"的记忆系统。

Claude Code 选择文件优先是因为它是开发工具，用户希望能直接看到和编辑记忆。如果它改成数据库后端，会失去这个核心优势。但它可以从其他系统借鉴不破坏文件优先性的设计，比如双时态字段、写时矛盾检测、SQLite FTS5 索引等。

四种系统代表了 AI 记忆研究的四个前沿：
  • Claude Code：开发者工效学（"记忆应该像代码一样可管理"）
  • Mem0：写时智能（"清洁的记忆库胜过聪明的检索"）
  • Letta：自主性（"代理应该管理自己的记忆"）
  • Zep：时态推理（"事实有时间，知识在演化"）
