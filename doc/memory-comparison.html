Memory 管理对比分析

Claude Code vs Memori — 两种截然不同的 AI 记忆架构哲学

⚙

1. 存储架构：文件 vs 数据库

Claude Code — 文件系统

~/.claude/projects/<path>/memory/
  MEMORY.md          # 索引文件（200行/25KB上限）
  user_role.md       # 记忆文件（frontmatter + 内容）
  feedback_test.md
  team/
    MEMORY.md        # 团队记忆索引
    project_auth.md
  • 人类可读，可直接编辑
  • 可 Git 版本控制
  • 零依赖，无需数据库
  • 检索依赖 LLM，不可结构化查询

Memori — SQL + 向量

SemanticTriple {
  subject_name, subject_type,
  predicate,
  object_name, object_type
}
Entity {
  facts: string[],
  fact_embeddings: vector[],
  semantic_triples: Triple[]
}
Memory { conversation, entity, process }
  • 结构化查询（SQL）
  • 向量相似度搜索（FAISS）
  • 语义三元组构建知识图谱
  • 需要数据库基础设施
启发：Claude Code 的文件方案对开发者友好（可读可编辑可版本控制），但缺乏结构化查询能力。可以考虑在文件系统之上增加一层轻量级索引（如 SQLite 或 JSON 索引），在不引入重基础设施的情况下实现更精确的检索。
✎

2. 记忆分类体系

Claude Code — 4 类型

user用户画像（角色、偏好、知识背景）
feedback行为反馈（纠正 + 确认）
project项目上下文（截止日期、决策、事故）
reference外部引用（Linear/Grafana/Slack 指针）

每种类型有明确的"何时保存"和"如何使用"指导。

Memori — 5 类别

fact事实性知识
preference用户偏好
skill技能/操作知识
context上下文信息
rule规则/约束

每种类别有不同的检索和保留行为。

启发：Memori 的 skill（技能）类别是 Claude Code 缺失的。Claude Code 的 feedback 类型部分覆盖了"如何做"的知识，但没有专门的操作性记忆（如"这个项目的部署流程是 X → Y → Z"）。另外 Memori 的 rule 类别也值得借鉴 — 明确的约束规则比混在 feedback 中更容易被系统强制执行。
★

3. 检索机制：LLM 判断 vs 混合搜索

Claude Code：LLM-as-Judge

  1. 扫描记忆目录，读取所有文件的 frontmatter（最多 200 个）
  2. 格式化为文本清单：[type] filename (时间): description
  3. 调用 Sonnet 3.5，传入用户查询 + 清单 + 最近使用的工具
  4. 模型返回最多 5 个最相关的文件名

优势 语义理解深，能处理模糊关联（"我上次说的那个 Bug" → 找到对应记忆）

劣势 每次检索消耗一次 API 调用；延迟高；200 文件上限是硬天花板

Memori：BM25 + FAISS 混合搜索

rank_score = (w_cos * dense_similarity) + (w_lex * lexical_score)

短查询（≤2 词）: w_lex = 0.30  // 更依赖精确词匹配
长查询（>2 词）:   w_lex = 0.15  // 更依赖语义相似度

候选池：有查询文本时扩大 10x（最少 50），让词汇信号影响排序

优势 毫秒级检索；可扩展到百万级记忆；无 API 调用成本；可离线运行

劣势 需要维护向量索引；对隐式关联的理解不如 LLM

启发：这是最值得 Claude Code 借鉴的点。当前的 LLM 检索方式在记忆文件增多时不可扩展（200 文件硬上限）。可以考虑两级检索：
  1. 第一级：轻量级向量/BM25 预筛选，从数百个记忆中快速缩小到 ~20 个候选
  2. 第二级：用 LLM 从候选中精选最终 5 个

这样既保留了 LLM 的深度理解，又突破了文件数上限，还减少了 LLM 需要处理的清单长度。

4. 记忆创建：子代理 vs API 拦截

Claude Code — 分叉子代理

  • runForkedAgent() 创建隔离子代理
  • 共享父级 Prompt 缓存，降低成本
  • 子代理用工具读取对话 → 写入记忆文件
  • 与主代理互斥：主代理写了就跳过后台
  • 最多 5 轮，60 秒超时

Memori — API 拦截 + 异步增强

  • 透明包裹任何 LLM 的 API 调用
  • 每轮对话自动触发 Memory Agent 提取
  • 异步增强管线：50 个并发 worker
  • 8 种增强类型：属性/事件/事实/人物/偏好/关系/规则/技能
  • 批量写入（100 条/批，0.1 秒超时）
  • 零延迟：提取完全在后台，不阻塞对话
启发：Memori 的"零延迟异步增强管线"是一个优秀设计。Claude Code 的分叉子代理虽然也是后台执行，但仍然消耗一次 API 调用。Memori 的 50-worker 批量写入模式在高频场景下效率更高。可以考虑将 Claude Code 的记忆提取从"调用 LLM 子代理"改为基于规则的快速提取 + 异步 LLM 精炼的两阶段模式。
☁

5. 生命周期管理

Claude Code — 渐进式压缩 + 合并

  • 微压缩：清理工具结果（实时）
  • 自动压缩：对话摘要化（会话内）
  • 会话记忆：12 分区结构化笔记（会话级）
  • 持久记忆：跨会话提取（永久）
  • Auto Dream：24h+5 会话后合并去重（跨会话）
  • 记忆新鲜度标记：>1 天附带警告

Memori — 重要性驱动的三级保留

  • 短期记忆：近期活动，~7 天过期
  • 长期记忆：重要信息、技能、偏好
  • 永久记忆：核心规则、关键偏好、基本事实
  • 创建时通过重要性评分决定保留层级
  • 无显式的合并/压缩机制
  • ID 缓存防止重复写入
启发：两种方案各有优势。Claude Code 的渐进式压缩更适合长对话场景（对话可能持续数小时），而 Memori 的重要性评分更适合高频短对话场景。一个值得探索的方向是结合两者：在创建时评分决定保留层级（如 Memori），同时在会话中做渐进式压缩（如 Claude Code）。另外，Memori 的 7 天短期过期机制是 Claude Code 缺少的 — 当前 Claude Code 的记忆没有自动过期，可能导致陈旧记忆堆积。

6. 上下文窗口策略

Memori 的双注入模式

Conscious Ingest（意识注入）

启动时一次性注入"本质身份上下文"，标记为 conscious-info，估算 100-300 token。类似于人类的"核心身份记忆"。

Auto Ingest（自动注入）

每次查询动态检索，上限 5 条记忆，估算 200-500 token。类似于人类的"按需回忆"。

两种模式合计平均 ~1,294 token/查询，仅占全上下文的 4.97%。

Claude Code 的动态管理

没有固定的"意识"/"自动"二分法，而是通过多层压缩动态管理：

  • 记忆注入：每文件 4KB，每轮 5 个 (~20KB)，会话累计 60KB
  • CLAUDE.md + 记忆文件作为用户上下文前置
  • 当上下文紧张时，通过压缩释放空间
启发：Memori 的"Conscious Ingest"概念非常有价值。将记忆分为"核心身份"（始终注入）和"动态回忆"（按需注入）是一个优雅的设计。Claude Code 可以考虑将某些 user 类型记忆标记为"核心"，保证它们始终在上下文中，而不是依赖每次的相关性选择。
★

7. 知识图谱：Claude Code 缺失的维度

Memori 的语义三元组

SemanticTriple {
  subject_name: "用户",     subject_type: "Person",
  predicate: "偏好使用",
  object_name: "TypeScript",  object_type: "Language"
}

// 可以推理：
// "用户" --偏好使用--> "TypeScript"
// "TypeScript" --被偏好使用于--> "项目A"
// 推导："用户" 在 "项目A" 中可能偏好 "TypeScript"

通过三元组构建实体间的关系网络，支持：

  • 关系推理：沿图谱路径发现隐式关联
  • 实体消歧：相同名称在不同上下文中的区分
  • 知识传播：一个实体的更新可以影响相关实体

Claude Code 的扁平结构

Claude Code 的记忆是独立的文件，文件之间没有显式的关联关系。相关性完全依赖 LLM 在运行时判断。

这意味着：

  • 无法通过结构化查询找到"与 X 相关的所有记忆"
  • 记忆之间的关联是隐式的（靠描述文本中的关键词）
  • 当记忆增多时，LLM 很难从 200 条描述中发现深层关联
启发：虽然完整的知识图谱对 Claude Code 的文件架构来说过重，但可以借鉴轻量级关联的思路。例如在 frontmatter 中增加 related 字段： 
---
name: auth-migration
type: project
related: [feedback_testing.md, reference_linear.md]
---

这样在检索时可以做"关联展开" — 选中一个记忆后，自动将其关联记忆也纳入候选。

8. 安全与多租户

Claude Code — 文件系统级隔离

  • 团队记忆通过目录前缀隔离
  • 路径遍历防护（5 种攻击模式）
  • 符号链接解析验证
  • 前缀攻击防护（要求分隔符）
  • 记忆文件权限：0o600（仅所有者读写）
  • 敏感数据警告（禁止 API Key 等）

Memori — 数据库级隔离

  • Entity/Process/Session 三级层次结构
  • 每条记忆必须标记 entity_id + process_id
  • 数据库级别的访问控制
  • 支持多租户 SaaS 部署
  • API Key 认证
各有所长：Claude Code 的文件系统安全防护非常细致（5 种路径遍历攻击、符号链接解析），适合本地开发场景。Memori 的数据库级隔离更适合多用户 SaaS 场景。对于 Claude Code 的团队功能演进，可以参考 Memori 的 Entity/Process 层级，在文件系统之上建立更细粒度的归属模型。
★

9. 关键启发总结

最值得借鉴的 5 个设计

#Memori 的设计对 Claude Code 的启发落地难度
1 混合检索（BM25 + 向量） 增加轻量级预筛选层，突破 200 文件上限，减少 LLM 调用 中 需引入嵌入模型或 SQLite FTS
2 Conscious Ingest（核心身份注入） 标记核心 user 记忆为"始终注入"，不依赖每次的相关性选择 低 在 frontmatter 加 priority: core 即可
3 三级保留（短/长/永久） 为记忆增加 TTL 机制，project 类型 30 天过期，user/reference 永久 低 基于 mtime + type 规则即可实现
4 语义三元组 / 关联关系 在 frontmatter 增加 related 字段，检索时做关联展开 低 不需要图数据库，文件引用即可
5 零延迟异步增强管线 将记忆提取拆为"快速规则提取" + "异步 LLM 精炼"两阶段 中 需要重构提取流程

Claude Code 的独有优势（不应丢失）

  • 文件可读可编辑 — 用户可以直接编辑记忆文件，Memori 的数据库做不到
  • Git 版本控制 — 记忆变更可追溯，可回滚
  • 渐进式压缩 — 对长对话场景的优化非常成熟（5 层压缩机制）
  • Auto Dream — 受生物学启发的记忆合并，这是 Memori 没有的
  • 零基础设施 — 不需要数据库、向量服务或任何外部依赖
  • 与 CLAUDE.md 深度集成 — 指令记忆和知识记忆统一在一个体系中
总体判断：Claude Code 和 Memori 代表了 AI 记忆管理的两种哲学 — "文件优先、LLM 驱动" vs "数据库优先、算法驱动"。最理想的方案是两者结合：保持 Claude Code 的文件可读性和零依赖特性，同时引入 Memori 的混合检索、核心记忆分层和自动过期机制。其中"两级检索"和"核心身份注入"是投入产出比最高的两个改进方向。
