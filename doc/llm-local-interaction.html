LLM 与本地交互全景

LLM 不能直接碰本地文件。它对本地的全部控制力，通过两条路径实现：被动接收（本地塞给它）和主动调用（它发 tool_use）

核心原则
LLM 是一个纯文本推理引擎，没有文件系统访问权。

它与本地的全部交互只有两条路径：

  被动  本地代码读好内容，塞进 API 请求 → LLM 打开"信封"就看到了
  主动  LLM 输出一段 JSON (tool_use) → 本地代码代替它执行 → 结果喂回去

路径 A: 被动接收 — 本地代码读好，塞进 API 请求
Step 0: 在 API 调用前，本地代码自动完成的所有读取 
本地 读 Context — getUserContext()    memoized, 会话内只执行一次
  ├ 遍历 CWD → 根目录，加载所有 CLAUDE.md 文件
  │   ├ /etc/claude-code/CLAUDE.md      (managed)
  │   ├ ~/.claude/CLAUDE.md              (user global)
  │   ├ ./CLAUDE.md, .claude/CLAUDE.md   (project)
  │   ├ .claude/rules/*.md               (project rules)
  │   └ CLAUDE.local.md                  (local, .gitignored)
  ├ 格式化 + 包裹 <system-reminder>
  └ → 注入 messages[0]（每轮 API 调用都带上）

本地 读 Context — getSystemContext()    memoized, 会话内只执行一次
  ├ git status + branch + recent commits (截断 2000 字符)
  └ → 追加到 system prompt 末尾（每轮 API 调用都带上）

本地 读 Memory 索引 — loadMemoryPrompt()    会话首次加载
  ├ 读取 ~/.claude/projects/<project>/memory/MEMORY.md
  ├ 截断到 200 行 / 25KB
  ├ 拼接记忆行为指令（4 种类型 + 保存/验证规则）
  └ → 注入 system prompt 的 memory section（每轮 API 调用都带上）

本地 + Sonnet 读 Memory 内容 — findRelevantMemories()    每轮异步
  ├ 本地 scanMemoryFiles() 扫描 memory/*.md (上限 200 个)
  ├ 本地 批量读取 frontmatter (name, description, type)
  ├ Sonnet sideQuery: 从候选中选 ≤5 个最相关记忆
  ├ 本地 读取选中文件内容 + 计算过期天数 + 过期警告
  └ → 注入下一轮 messages 附件 (system-reminder)

本地 读 Tool Descriptions — toolToAPISchema()
  ├ 43 个工具的 tool.prompt() 生成 description
  ├ Zod schema → JSON Schema
  └ → 注入 API 请求的 tools 参数

LLM 收到请求时，这些内容已经"在那了"，它不需要做任何事就能看到
路径 A 注入后，LLM 看到的完整请求
system: [
  静态行为指令      ← 角色/风格/安全/工具规则 (scope: global)
  记忆指令+MEMORY.md索引 ← 4 种类型定义 + 已有记忆目录
  环境信息            ← cwd, shell, OS, model, date
  git status          ← branch, status, recent commits
]

messages: [
  messages[0]: CLAUDE.md  ← 项目指令+全局指令, 包裹在 <system-reminder>
  上轮记忆附件          ← Sonnet 选出的 ≤5 个记忆文件内容 (如果有)
  用户消息...
  助手回复...
  tool_result...
]

tools: [
  43 个工具定义  ← name + description + input_schema
]
↓ LLM 开始推理，可能决定调用工具 ↓
路径 B: 主动调用 — LLM 发 tool_use，本地代替执行
Step 1-7: tool_use → 本地执行 → tool_result 循环 
Step 1: LLM 输出 tool_use
LLM 流式返回 assistant message，含 tool_use content block:
  { type: "tool_use", id: "toolu_01...", name: "Read", input: { file_path: "/tmp/a.ts" } }

  流式组装过程 (SSE):
  ├ content_block_start  → 创建骨架 { id, name, input: "" }
  ├ content_block_delta  → input += partial_json (逐片拼接)
  └ content_block_stop   → JSON.parse(input) → 完整对象

Step 2: 本地提取
本地 query.ts:829
  ├ content.filter(c => c.type === 'tool_use')
  └ 一条 assistant message 可能含多个 tool_use (并行调用)

Step 3: 输入验证
本地 toolExecution.ts
  ├ inputSchema.safeParse(input)  ← Zod schema 验证
  ├ tool.validateInput()          ← 工具自定义验证 (可选)
  └ 失败 → 返回 error tool_result，LLM 收到错误信息

Step 4: 权限检查
本地 toolExecution.ts + permissions.ts
  ├ PreToolUse hooks   ← 用户自定义 shell 命令拦截
  ├ hasPermissionsToUseTool()
  │   ├ alwaysAllow rules  → 直接放行
  │   ├ alwaysDeny rules   → 直接拒绝
  │   └ alwaysAsk rules    → 弹出对话框等用户确认
  └ 拒绝 → error tool_result 返回给 LLM，LLM 调整策略

Step 5: 本地执行
本地 tool.call(input, context)
  ├ 调度: isConcurrencySafe() ? 并发(上限10) : 串行
  └ 实际操作:
      Read   → fs.readFile()          读文件内容
      Write  → fs.writeFile()         创建/覆写文件
      Edit   → 字符串替换后写回       精确编辑
      Bash   → child_process.exec()   执行 shell 命令
      Grep   → ripgrep 搜索           文件内容正则搜索
      Glob   → 文件名模式匹配         查找文件
      Agent  → fork QueryEngine       启动子 Agent
      MCP    → MCP Server 调用        外部工具协议

Step 6: 构造结果
本地 toolExecution.ts
  ├ tool.mapToolResultToToolResultBlockParam(data, toolUseID)
  │   → { type: "tool_result", tool_use_id: "toolu_01...",
  │        content: "文件内容...", is_error: false }
  ├ applyToolResultBudget()
  │   → 超过 maxResultSizeChars? 落盘，只发预览+文件路径
  └ PostToolUse hooks ← 可修改结果

Step 7: 喂回 LLM
本地 query.ts
  ├ tool_result 加入 messages 数组
  ├ 下一次 API 调用发送
  └ LLM 看到结果:
      ├ stop_reason == 'tool_use'   → 回到 Step 1（继续循环）
      └ stop_reason == 'end_turn'   → 完成回答

主动 Web (WebSearch/Fetch) tool_result 在 messages 中 LLM 按需 每次 +0.5-3% 网页内容可能很长
主动 MCP 工具 tool_result 在 messages 中 LLM 按需 不定 外部工具服务器返回结果

记忆的读与写分别走哪条路径

记忆系统横跨两条路径 — 读走被动，写走主动

记忆操作路径
路径 操作 执行者 过程
被动 读 MEMORY.md 索引 本地 会话启动时读取，注入 system prompt。LLM 每轮都能看到索引
被动 选择相关记忆 Sonnet 每轮异步 sideQuery，从 frontmatter 列表中选 ≤5 个文件
被动 读记忆文件内容 本地 读取 Sonnet 选中的文件，截断 + 过期警告，注入下一轮附件
主动 写记忆文件 LLM LLM 发 tool_use: Write → 本地写 memory/*.md
主动 更新 MEMORY.md 索引 LLM LLM 发 tool_use: Edit → 本地编辑 MEMORY.md 追加条目
主动 验证记忆准确性 LLM LLM 发 tool_use: Grep/Read → 确认记忆中的文件/函数是否仍存在
被动 后台提取 本地 + LLM 轮次结束后 fork Agent，分析对话，自动写入新记忆（受限权限）

Context% 增长归因

HUD 的 Context% 增长时，哪条路径贡献了多少

增长归因
路径 A (被动) 贡献的 Context%：~10% 固定，不增长
  ├ System Prompt (静态+动态)   ~3-4%
  ├ Tools Schema               ~5%
  ├ CLAUDE.md                   ~1-2%
  ├ git status                  <0.5%
  ├ MEMORY.md 索引+指令         ~1-2%
  └ 相关记忆附件                0-2% (每轮波动，不累积)

路径 B (主动) 贡献的 Context%：持续增长，是增长主力
  ├ tool_result 累积            最大变量，每次 +0.1-5%
  ├ LLM 回复 (text+thinking)    每轮 +0.1-2%
  └ 用户消息                    每轮 +0.05-0.3%

Compact 压缩效果：
  └ 路径 B 的累积全部清零 → Context% 从 ~85% 降到 ~12%
    路径 A 的固定开销保留（重新注入）
一句话总结：
LLM 对本地的全部交互分两条路径。路径 A（被动）是本地代码把 CLAUDE.md、git status、Memory 索引和相关记忆塞进 API 请求 — 这是固定开销，占 ~10%，不增长。路径 B（主动）是 LLM 通过 tool_use 驱动本地执行文件读写、命令、搜索 — 这是 Context% 增长的主力，也是 compact 压缩的目标。两条路径共同构成了 LLM 的全部本地感知和操控能力。

其他模型要复用这套系统，需要哪些能力

API 格式可以用 Shim 转译，但模型本身的推理能力无法绕过

能力分层：可适配 vs 不可复制
难度 能力 可否 Shim 说明
可以通过 Shim/适配层解决（API 格式层面）
易 Tool Use 格式 可以 Anthropic tool_use ↔ OpenAI function_call 双向转换。openclaude 的 openaiShim.ts 已实现
易 流式事件格式 可以 OpenAI SSE → Anthropic SSE 转译
易 消息/System Prompt 格式 可以 system 参数 → messages 合并, content block 结构差异
易 Stop Reason / Usage 映射 可以 end_turn→stop, tool_use→tool_calls 等
中 Prompt Cache 降级可用 其他 Provider 有不同缓存机制或没有。忽略 cache_control 即可，只是更贵更慢
中 Token 计数 API 降级可用 回退到启发式估算 (4 bytes/token)，上下文管理精度下降
中 Extended Thinking 降级可用 没有 thinking 块。部分模型有 reasoning 输出 (o1/o3) 但格式不同
需要模型本身具备的推理能力（不可 Shim，取决于模型智力）
硬 工具选择判断力 不可 从 43 个工具中选对的。何时用 Read vs Grep，何时 spawn Agent vs 直接搜索。需要语义理解
硬 并行 vs 串行决策 不可 哪些工具调用可并行，哪些有因果依赖。"Read before Edit" 等规则需要逻辑推理
硬 权限拒绝后的适应 不可 被拒绝后不能盲目重试，必须推理原因并换方案。需要 theory of mind
硬 错误诊断能力 不可 工具返回错误时，区分用户错误/工具错误/模型错误，诊断根因后再重试
硬 50-80 条规则同时遵守 不可 system prompt 含 50+ 条并行约束（安全/风格/git/注释/scope），需同时跟踪上下文化应用
硬 代码风格匹配 不可 读现有代码 → 匹配缩进/命名/导入风格 → 生成一致代码。不是套模板能解决的
硬 安全意识 不可 写代码时主动避免注入/XSS/CSRF，发现写了不安全代码立即修正
硬 记忆判断力 不可 判断何时保存记忆、保存什么类型、是否重复、是否需要验证。需要语用学推理
硬 Compact 摘要质量 不可 9 段结构化摘要需要: 保留精确代码片段、文件名行号、用户反馈演变、决策理由
硬 Scope 自律 不可 "不要添加未请求的功能" "不要改进相邻代码" — 需要克制扩展冲动的能力
硬 诚实报告 不可 测试失败时如实报告，不声称成功。区分已验证 vs 未验证。不模糊确定结果
需要 Anthropic 特有基础设施（第三方无法替代）
锁定 Sonnet sideQuery（记忆选择） 需替换 用 Sonnet 小模型选相关记忆。可用其他小模型替代，但选择质量取决于模型能力
锁定 后台记忆提取 Agent 需替换 fork Agent 自动从对话提取记忆。可用其他模型，但提取质量取决于推理能力
锁定 12 个 Beta Features 不可 interleaved-thinking, context-1m, fast-mode, tool-search 等。纯 Anthropic 专有

结论：
API 格式差异可以用 Shim 解决（openclaude 已证明）。但模型的推理质量无法绕过：
• 工具选择需要从 43 个中精准选择 + 判断并行/串行
• 50+ 条规则同时遵守需要强 instruction following
• 记忆判断需要知道什么值得记、什么不值得
• Compact 摘要需要保留精确代码片段和决策理由
• Scope 自律需要克制"改进"冲动 — 这可能是最难复制的
小模型可以通过 Shim 调通 API，但实际使用体验会显著下降。系统设计假设的是顶级模型的推理能力。
