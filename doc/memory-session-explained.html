Claude Code 记忆与 Session 深度解析

从真实使用场景出发，理解 Session、Memory、Context 如何协作，跨会话记忆如何流转，系统能做什么和不能做什么

一、三个核心概念

概念一句话存在哪里生命周期类比
Session 一次对话进程 进程内存 + jsonl 启动 → 退出 打开文档编辑，关掉就没了（但有自动保存）
Memory 跨会话的持久知识 磁盘 .md 文件 永久 笔记本上的笔记，下次打开还在
Context 项目级的固定指令 CLAUDE.md + git status 会话级缓存 办公室墙上的规章制度
注意：claude-hud 的 "Context" 是第四个概念

claude-hud 状态栏显示的 Context% 和上面的 Context 不是同一个东西：

概念含义包含什么
Context
（我们讨论的）		 项目级固定指令
CLAUDE.md + git status		 只是 API 请求中的一个组成部分，占 ~1-2%
HUD Context%

HUD Context% 的组成：

  System Prompt     ~3-4%  固定  (含 Memory 指令 + MEMORY.md 索引)
+ Tools Schema      ~5%    固定  (43 个工具定义)

+ Memory 附件       0-2%   波动  (Sonnet 选出的记忆内容)
+ 对话历史          持续增长      (用户消息 + LLM 回复)

─────────────────────────────────
= HUD Context%    / 200K 或 1M 窗口

Context 和 Memory 索引是固定开销（~10%），不是增长源。
tool_result 是增长主力 — 一次 Read 大文件就可能 +5%。
接近上限时 auto-compact 触发，Context% 从 ~85% 降到 ~12%。

二、真实场景：跨 Session 的记忆流转

以 /opt/mywork/mytest/bl（Blender MMD 转换插件，37 个记忆文件）为例

Session 1：工作 + 保存状态
你: "帮我记录一下，下次重启能接着干"

Claude 做了三件事（都是 tool_use 主动调用）:

• Write → Convert_to_MMD_claude/doc/hip_leg_fix_methods.md (170 行)
  给人看的完整文档，commit 到 git

• Write → memory/project_hip_leg_fix_methods.md (70 行)
  给 LLM 自己看的浓缩版（诊断顺序 + 方法 + 踩坑）

• Edit → memory/MEMORY.md
  追加索引: "[腿胯修复方法清单] — L4→L1→L2→L3诊断"

• 系统自动: session jsonl 已在实时写入
  完整对话流水账，每条消息逐行追加
两个文件的关系: memory/project_hip_leg_fix.md (70行) 和 doc/hip_leg_fix.md (170行) 之间没有显式关联。它们是 LLM 在同一次对话中分别独立写的两个文件。记忆文件不引用文档路径，文档也不知道记忆存在。后续如果改了文档没改记忆（或反过来），系统不会报警。
↓ 退出 Claude ↓
Session 2：新会话，靠记忆接力
启动阶段（自动，你不做任何事）:
  Context 读 CLAUDE.md → messages[0]           ✓ 每轮可见
  Context 读 git status → system prompt 末尾  ✓ 每轮可见
  Memory  读 MEMORY.md 索引 → system prompt    ✓ 每轮可见 (36 行标题)
  Session 上次的 jsonl?                          ✗ 没人读

你发第一条消息: "接着腿部问题继续"
  Memory  Sonnet 异步启动 → 选中 hip_leg_fix
                                                Δ 内容在下一轮才注入

LLM 第 1 轮:
  只看到 MEMORY.md 索引标题 (~5% 信息)

LLM 第 2 轮:
  看到记忆内容 70 行浓缩版 (~15% 信息)
  如果需要更多 → tool_use: Read doc/hip_leg_fix.md
  现在看到完整 170 行 (~75% 信息)
信息漏斗
Session 1 对话全文              100%
  ├ jsonl (自动保存)             100%  但下次不自动读
  ├ doc/*.md (LLM 写)            ~60%  需 LLM 用 Read 主动读
  └ memory/*.md (LLM 写)         ~30%
       ├ MEMORY.md 索引           ~5%   每轮自动可见
       └ Sonnet 选 ≤5 个          ~10%  下一轮才可见，不一定被选中

Session 2 第一轮 LLM 看到的:    ~5%
Session 2 第二轮 LLM 看到的:    ~15%
如果 LLM 主动 Read 了 doc:     ~75%

三、LLM 与本地交互的两条路径

维度路径 A: 被动接收路径 B: 主动调用
谁驱动本地代码自动LLM 发 tool_use
内容CLAUDE.md, git status, MEMORY.md, 相关记忆任意文件、命令、搜索
LLM 感知不知过程，只看到结果自己决定的
权限无（系统内部）有（hooks → rules → 确认）
时机API 调用前API 调用后
Context% 影响~10% 固定tool_result 累积，增长主力

记忆横跨两条路径 — 读走被动，写走主动：

路径操作说明
被动读 MEMORY.md 索引启动时注入 system prompt
被动Sonnet 选择相关记忆每轮异步 sideQuery
被动读记忆文件内容选中的文件注入下一轮附件
主动写记忆 / 更新索引LLM 发 tool_use: Write/Edit
主动验证记忆准确性LLM 发 tool_use: Grep/Read 确认
被动后台提取fork Agent 自动分析对话写入

四、系统能做什么和不能做什么

能做的
每轮自动展示记忆索引MEMORY.md 注入 system prompt
每轮智能选择相关记忆Sonnet sideQuery 选 ≤5 个
LLM 主动保存记忆识别纠正/确认/项目信息后写入
后台自动提取记忆fork Agent 分析对话 (feature flag)
记忆过期警告>1天 的记忆附带过期提示
按项目隔离每个 git root 独立 memory/
Session 完整持久化jsonl 实时写入，--resume 恢复
记忆提升到 CLAUDE.md/remember 技能
不能做的
不会自动读历史 session jsonl上次对话的完整信息静躺磁盘，除非 --resume
不会自动从文档生成记忆doc/*.md 和 memory/*.md 独立写入，无同步
不会检测记忆与文档漂移改了文档没改记忆（或反过来）不报警
不会补回遗漏的记忆Session 1 漏记了，Session 2 无法发现
不保证记忆质量取决于 LLM 当时的判断力
不保证记忆被选中37 个中每轮选 5 个，命中率 ~14%
会话内不刷新 ContextCLAUDE.md memoized，需 /compact 刷新
索引更新不原子LLM 可能写了文件但忘了更新 MEMORY.md

六、数量限制

限制项上限超限后果
MEMORY.md 索引行数200 行第 201 行截断，LLM 不知道这些记忆存在
MEMORY.md 字节数25 KB超限截断，追加警告
记忆文件扫描数200 个第 201 个文件不进候选
每轮选择数≤5 个200 个文件也只读 5 个内容
项目隔离每个 git root换目录 = 另外 200 行
过期警告>1 天附带过期提示，LLM 被要求先验证
200 行限制是 token 预算：200行 × ~15 token/行 ≈ 3,000 tokens 固定开销。不限制则 1000 条 = 15,000 tokens，光索引就占 200K 窗口的 7.5%。

七、提高记忆可靠性的实践

优先级做法原理
必做 关键交接信息写到 CLAUDE.md 每轮必定可见，不依赖 Sonnet 选择
必做 退出前检查记忆写入成功 "看一下你刚写的 memory，索引更新了没"
推荐 索引里加"必读"标签 提高 Sonnet 选中概率（提示词工程，非系统机制）
推荐 启动时主动提醒 "先读 memory 里的 handoff 和 doc/xxx.md"
可选 重要场景用 --resume 信息零丢失，代价是 Context% 从历史 token 开始
可选 定期 /remember 审查 把重要记忆提升到 CLAUDE.md

八、概念关系图

                 ┌───────────────────────────────────────┐
                 │         Session (进程内存)            │
                 │  对话消息 ←→ LLM 推理 ←→ 工具执行  │
                 └────┬──────────┬──────────┬────────────┘
                      │          │          │
               自动实时写    LLM 主动写   LLM 主动写
                      │          │          │
               ┌──────↓──┐ ┌─────↓────┐ ┌──↓──────────┐
               │ jsonl    │ │ memory/   │ │ doc/*.md    │
               │ 原始录像 │ │ 浓缩笔记  │ │ 完整报告    │
               │ 100%    │ │ ~30%      │ │ ~60%        │
               └──────┬──┘ └─────┬────┘ └──┬──────────┘
                      │          │          │
               --resume   ┌──────┘          │
               手动恢复   │ 自动            │ LLM 用 Read
                      │   │                │ 主动读取
               ┌──────↓───↓────────────────↓──────────┐
               │      下一次 Session                  │
               │                                        │
               │  system prompt: MEMORY.md 索引 (被动)  │
               │  messages[0]: CLAUDE.md (被动)         │
               │  附件: Sonnet 选的记忆 (被动)          │
               │  tool_result: Read doc/ (主动)         │
               │                                        │
               │  jsonl? 不读 (除非 --resume)        │
               └────────────────────────────────────────┘
一句话总结：Session 是信息的唯一来源，Memory 和 Doc 都从它提炼。但下次 Session 只自动读 Memory（被动路径），不读历史 jsonl。写入记忆的质量决定了跨 Session 的信息保真度 — 而这取决于 LLM 的判断力，是整个系统最脆弱也最不可 Shim 的环节。
