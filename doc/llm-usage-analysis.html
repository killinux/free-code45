LLM 驱动本地 CLI 能力（Agent 工具链）

LLM 通过 tool_use 调用本地能力的完整链路 — 这是整个系统的"Agent"核心

完整执行链路
1. LLM 返回 assistant message，含 tool_use content block (name + input)
   ↓
2. query.ts 提取 ToolUseBlock[]
   ↓
3. toolOrchestration.ts → runTools()：按并发安全性分批
   ↓
4. toolExecution.ts → runToolUse()：单工具执行入口
   ↓
5. 权限检查：PreToolUse hooks → canUseTool() → hasPermissionsToUseTool()
   ↓ (允许)
6. tool.call(input) → 执行本地操作（Bash/文件IO/搜索...）
   ↓
7. ToolResult → mapToolResultToToolResultBlockParam() → tool_result block
   ↓
8. tool_result 加入消息历史，循环回到 step 1（直到 stop_reason = end_turn）
tool_use Content Block 详解

LLM 返回的 assistant message 中，content 数组包含多种 block 类型。tool_use 是驱动本地能力的核心。

assistant message 的 content 数组（可包含多个 block 交替出现）
content: [
  { type: "text",      text: "Let me check the files." },
  { type: "tool_use",  id: "toolu_01...", name: "Bash", input: { command: "ls" } },
  { type: "tool_use",  id: "toolu_02...", name: "Read", input: { file_path: "/tmp/a.ts" } },
  { type: "thinking",  thinking: "I should...", signature: "..." }  // 仅 thinking 启用时
]

     ↓

     ↓

     ↓

   • hasPermissionsToUseTool() → allow (匹配 alwaysAllowRules)
     ↓

     ↓

     ↓

     ↓

关键设计原则：
• System Prompt 是"行为定义"— 告诉 LLM 怎么做（角色、风格、安全准则）
• User Context 是"知识注入"— 告诉 LLM 知道什么（CLAUDE.md、项目规则）
• System Context 是"环境状态"— 告诉 LLM 在哪里（git branch、working dir）
• system-reminder 是"动态补充"— 告诉 LLM 当前情况（记忆、工具列表、警告）
• Tool Descriptions 是"能力声明"— 告诉 LLM 能做什么（工具名称、参数、用法）
缓存策略的核心思路：静态内容（所有用户相同的行为指令）用 global scope 跨 org 共享；动态内容（记忆、环境、MCP）按会话独立。

      CLAUDE.md 通常由人工维护，LLM 仅在用户明确要求时修改
设计总结：
• Context 是"基础设施"— 加载一次，贯穿全场。CLAUDE.md 是项目的"宪法"，git status 是"当前地图"
• Memory 是"工作记忆"— 每轮动态选择最相关的片段。MEMORY.md 索引是"目录页"，主题文件是"详细内容"
• 时序关键：Memory prefetch 是异步的，结果注入到下一轮。这意味着 LLM 第一次提到某个记忆时，只有索引可用（来自 MEMORY.md）；完整内容要等到下一轮才可见
• 不重叠：Context 在 system prompt / messages[0]；Memory 内容在 message 附件。二者注入位置不同、刷新机制不同、写入方不同

Prompt Cache（API 服务端缓存）

人们常说的"Claude 缓存"指的是 Anthropic 的 Prompt Caching — 服务端缓存重复发送的 prompt 前缀，降低费用和延迟。不减少 Context%，只减少账单。

缓存 scope适用内容TTL说明
scope: 'global' 静态 System Prompt（6 段） >5 分钟 所有用户共享（相同 CLI 版本的静态行为指令完全一样）
scope: 'org' System Prompt Prefix >5 分钟 同组织内共享
ephemeral Messages 中的缓存锚点 5 分钟 / 1 小时 对话前缀不变时命中。1h TTL 通过 feature flag 控制
null 动态 System Prompt、新消息 — 不缓存
缓存经济学
典型 API 调用 (Opus, 第 5 轮):

总 input: 50,000 tokens
  ├ cache_read:     30,000 tokens  (system prompt + tools, 命中缓存)
  ├ cache_creation:      0 tokens  (已缓存, 无需创建)
  └ 真正计算的 input: 20,000 tokens  (新对话历史 + tool results)

有缓存: 30,000 × $0.50/Mtok + 20,000 × $5/Mtok = $0.115
无缓存: 50,000 × $5/Mtok                       = $0.250
节省 54%

什么操作会破缓存 (cache break):
  • MCP 指令变化           → 破  (DANGEROUS_uncached 改变 system prompt)
  • /compact               → 破  (messages 前缀全变)
  • 新会话                 → 破  (但 global scope 静态 prompt 仍可命中)
  • 切换模型               → 破  (不同模型缓存独立)
  • 用户发新消息           → 不破 (追加在末尾, 前缀不变)
  • 工具执行返回结果       → 不破 (追加在末尾)
代码中的 4 种 "缓存" 概念区分
缓存 层级 影响 说明
Prompt Caching API 服务端 费用 + 延迟 大家说的 "Claude 缓存"。cache_control 标记，服务端缓存 prompt 前缀
getUserContext() memoize 本地内存 CLAUDE.md 加载次数 会话内只读一次 CLAUDE.md，/compact 刷新
getToolSchemaCache() 本地内存 工具 schema 稳定性 防止 GrowthBook 中途改 flag 导致 schema 变化
getMemoryFiles() memoize 本地内存 记忆文件扫描次数 会话内只扫一次 memory/ 目录，/compact 刷新
Compact 与 Cache 的关系：
• /compact 必然破缓存 — messages 前缀被摘要替换，服务端 Prompt Cache 失效
• 但 compact 后的下一次 API 调用会重建缓存（system prompt + tools 写入新缓存）
• compact 用的 LLM 调用本身可以复用主线程的 prompt cache（fork agent 共享缓存前缀）减少摘要成本
• Tier 2 Microcompact 的 cached 变体使用 cache_edits API，在不破缓存的前提下清理旧 tool_result
