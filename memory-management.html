Architecture Overview

The memory management system consists of 5 layers working in concert, from real-time to persistent:

LayerPurposeCostPersistence
Token Metering Measure context window utilization Zero (calculation only) Per-request
Micro Compact Lightweight tool result eviction Zero / cache_edits API Per-session
Auto Compact Full conversation summarization 1 API call (legacy path) Per-session
Session Memory Structured session notes for fast compaction Forked agent (cache-shared) Per-session file
Auto Memory Cross-session durable knowledge Forked agent (cache-shared) Permanent files

1. Token Metering

src/utils/tokens.ts

Core Function: tokenCountWithEstimation()

This is the canonical function for measuring context size. All threshold checks (auto-compact, session memory) use it.

Algorithm

  1. Find the last assistant message with valid usage data
  2. Compute full context: input_tokens + cache_creation + cache_read + output_tokens
  3. Walk back to the first sibling with the same message.id (handles parallel tool calls)
  4. Add rough estimate for messages after that anchor point
Parallel Tool Call Handling: When the model issues multiple tool calls, streaming emits separate assistant records sharing one message.id. The function walks back to the first sibling to ensure interleaved tool_results are counted.

Related Functions

FunctionReturnsUse Case
getTokenCountFromUsage() Full context size from single Usage object Building block for other functions
tokenCountFromLastAPIResponse() Full context from last API call Pre-compact baseline measurement
finalContextTokensFromLastResponse() input + output (no cache) Task budget remaining computation
messageTokenCountFromLastAPIResponse() Output tokens only Measuring single response size (NOT for thresholds)

2. Micro Compact

src/services/compact/microCompact.ts

The lightest form of context reclamation. Runs before every API call.

Path A: Time-Based Trigger

Direct Mutation

When the time gap since the last assistant message exceeds the threshold, the server cache has expired anyway. Directly clears old tool result content.

  • Replaces content with [Old tool result content cleared]
  • Keeps the most recent N tool results (floor at 1)
  • Resets cached MC state since server cache is invalidated

Path B: Cached Microcompact

Cache Edits API

When cache is warm, uses the cache_edits API to delete tool results server-side without invalidating the cached prefix.

  • Does NOT modify local message content
  • Count-based trigger/keep thresholds from GrowthBook
  • Tracks tool results via CachedMCState
  • Queues pendingCacheEdits for the API layer

Compactable Tools

Only these tool types are eligible for microcompaction:

FileRead Bash Grep Glob WebSearch WebFetch FileEdit FileWrite

3. Auto Compact

src/services/compact/autoCompact.ts

Threshold Calculation

effectiveWindow = min(modelWindow, CLAUDE_CODE_AUTO_COMPACT_WINDOW) - 20K
autoCompactThreshold = effectiveWindow - 13K   // ~93% occupancy
blockingLimit = effectiveWindow - 3K           // ~98%, blocks user input
13K AUTOCOMPACT_BUFFER_TOKENS
20K WARNING / ERROR threshold buffer
3K MANUAL_COMPACT_BUFFER (blocking limit)
20K MAX_OUTPUT_TOKENS_FOR_SUMMARY

Execution Priority

  1. Session Memory Compact (zero API cost) — uses pre-extracted session notes as summary
  2. Legacy Compact (1 API call) — sends full conversation to LLM for summarization

Post-Compact Cleanup

  • Resets lastSummarizedMessageId (old UUIDs no longer exist)
  • runPostCompactCleanup() restores skills and CLAUDE.md files (5KB/skill, 25K total budget)
  • Notifies prompt cache break detection to expect cache read drop

Circuit Breaker

MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3
After 3 consecutive failures, auto-compact stops retrying for the rest of the session. Background: 1,279 sessions had 50+ consecutive failures (up to 3,272), wasting ~250K API calls/day globally.

4. Session Memory

src/services/SessionMemory/sessionMemory.ts

Maintains a structured Markdown file with notes about the current conversation. Runs periodically via a post-sampling hook.

Trigger Thresholds

ParameterDefaultDescription
minimumMessageTokensToInit 10,000 Minimum tokens before first extraction
minimumTokensBetweenUpdate 5,000 Token growth required between extractions
toolCallsBetweenUpdates 3 Minimum tool calls between extractions

Trigger Logic

shouldExtract = (tokenThreshold AND toolCallThreshold)
             OR (tokenThreshold AND noToolCallsInLastTurn)

Token threshold is always required — even if tool call count is met, extraction waits for token growth.

Execution Mechanism

  1. Check feature gate tengu_session_memory (cached, non-blocking)
  2. Create isolated subagentContext to avoid polluting parent cache
  3. Set up session memory file at ~/.claude/session-memory/MEMORY.md
  4. Read current memory content via FileReadTool
  5. Build extraction prompt with buildSessionMemoryUpdatePrompt()
  6. Run runForkedAgent() with querySource: 'session_memory'
  7. Agent can ONLY use FileEdit on the exact memory file path

Memory File Structure (12 Sections)

Session Title, Current State, Task Specification, Files/Functions, Workflow, Errors & Corrections, System Documentation, Learnings, Key Results, Worklog, etc.

  • Max 2K tokens per section
  • 12K tokens total limit
  • Truncated by truncateSessionMemoryForCompact() before use in compaction

5. Session Memory Compact

src/services/compact/sessionMemoryCompact.ts

Uses existing session memory as the summary instead of calling the API. Zero API cost.

Message Preservation Config

ParameterDefaultDescription
minTokens10,000Minimum tokens to preserve after compaction
minTextBlockMessages5Minimum messages with text blocks to keep
maxTokens40,000Hard cap on preserved tokens

calculateMessagesToKeepIndex()

  1. Start from lastSummarizedMessageId + 1
  2. Expand backwards until meeting BOTH minimums (tokens AND text-block messages)
  3. Stop if hitting maxTokens cap
  4. Floor at the last compact boundary marker
  5. Call adjustIndexToPreserveAPIInvariants()

adjustIndexToPreserveAPIInvariants()

Ensures no tool_use/tool_result pairs are split and thinking blocks sharing the same message.id are included.

Bug scenario it fixes: Streaming emits separate records per content block with same message.id. If the slice boundary lands between split records, normalizeMessagesForAPI merges them and creates orphan tool_result references → API error.

Two Scenarios

Normal Case

lastSummarizedMessageId is set. Keep only messages after that ID, expanded to meet minimums.

Resumed Session

lastSummarizedMessageId is not set but session memory has content. Keep all messages but use session memory as summary.

6. Auto Memory (Persistent)

src/services/extractMemories/extractMemories.ts

Extracts durable memories from the session and writes them to ~/.claude/projects/<path>/memory/. Runs at the end of each complete query loop.

Trigger Conditions

  • Last assistant turn has no tool calls (natural conversation break)
  • Called via handleStopHooks()
  • Only for the main agent (not subagents)
  • Throttled: every N eligible turns (tengu_bramble_lintel, default 1)

Execution Design

Mutual Exclusion

If the main agent already wrote to memory files this turn, the background extraction is skipped and the cursor advances past the range.

Overlap Protection

If a new extraction request arrives while one is in-progress, the context is stashed. After the current run finishes, one trailing extraction runs with the latest stashed context.

Resource Limits

  • maxTurns: 5 — hard cap prevents verification rabbit-holes
  • drain timeout: 60s — drainPendingExtraction() waits before graceful shutdown
  • skipTranscript: true — avoids race conditions with main thread

Permission Model

createAutoMemCanUseTool() defines a strict sandbox:

ToolPermission
FileRead, Grep, GlobUnrestricted (read-only)
BashRead-only commands only (ls, find, grep, cat, stat, wc, head, tail)
FileEdit, FileWriteMemory directory only
REPLAllowed (inner primitives still gated)
All othersDenied

8. Safety Guards

Recursion Prevention

These querySource values are blocked from triggering auto-compact:

  • session_memory — would deadlock (forked agent)
  • compact — would deadlock (forked agent)
  • marble_origami — context-collapse agent, module-level state conflict

Circuit Breaker

  • After 3 consecutive auto-compact failures, stops retrying
  • Prevents API hammering when context is irrecoverably over limit
  • Resets on success

API Invariant Protection

  • tool_use / tool_result pairs are never split
  • Thinking blocks sharing message.id are always included together
  • Compact boundary markers filtered from messagesToKeep to prevent double-prune

Mutual Exclusion

  • Session Memory extraction uses sequential() wrapper
  • Auto Memory skips if main agent wrote memory this turn
  • Overlap: stash context, run one trailing extraction
  • Context Collapse and Reactive Compact disable proactive auto-compact
